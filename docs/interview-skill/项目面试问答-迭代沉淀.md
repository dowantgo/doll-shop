## 1. 介绍一下这个项目？【已实现】
### 场景背景
希望做一个包含“完整交易闭环”的项目，不是只做商品展示，而是覆盖从用户认证到支付完成的完整链路。

### 方案选择
采用前后端分离：前端负责交互体验，后端负责业务规则和数据一致性。

### 对比权衡
- 一体化模板开发速度快，但前后端职责耦合，不利于扩展。
- 分离架构联调成本更高，但更接近真实团队协作模式。

### 实现路径
- 前端：路由守卫 + API 分层 + 统一请求拦截。
- 后端：按业务域拆 app（users/products/cart/orders/payment）。
- 支付：创建支付单、轮询状态、异步回调验签。

### 效果指标
- 能完整演示核心电商链路。
- 能支持管理端运营动作（商品、订单、用户管理）。

### 风险边界
- 支付回调依赖公网可达地址。
- 高并发能力仍需二迭代强化。

### 可扩展点
二迭代接入评价、榜单、热榜轮播、性能优化和容器化。

## 6. xx 场景下，A 和 B 怎么选？【已实现 + 规划中】
### 场景样例 1：缓存 vs 数据库直查
- 读多写少热点（热榜、排行）优先缓存。
- 强一致写场景（扣库存、支付状态）必须以数据库事务为准。

### 场景样例 2：同步调用 vs 异步队列
- 用户即时反馈场景（下单响应）先同步最小闭环。
- 非核心链路（日志、推荐刷新）可异步队列化（迭代2/3规划）。

### 场景样例 3：乐观锁 vs 悲观锁
- 高冲突库存扣减优先悲观锁/原子扣减（当前已采用事务锁思路）。
- 冲突低可考虑乐观锁提升吞吐。

## 7. 项目难点是什么？【已实现】
### 难点 1：支付链路一致性
支付涉及第三方接口、回调验签、订单状态变更，容易出现“支付成功但订单未更新”。

### 难点 2：环境联调复杂
本地、ngrok、反向代理、支付回调等多环境切换容易出现配置漂移。

### 难点 3：事务边界
`select_for_update` 必须在事务中使用，否则会报错并引发 500。

### 我怎么解决
- 明确支付状态机与回调处理逻辑。
- 统一环境变量入口并规范启动手册。
- 在关键写路径上显式加事务。

```python
with transaction.atomic():
    cart_items = get_user_cart(user)
    if not cart_items:
        return 400
    for item in cart_items:
        product = Product.objects.select_for_update().get(id=item.product_id)
        if product.stock < item.qty:
            raise BizError("库存不足")
    order = create_order(user, address)
    for item in cart_items:
        create_order_item(order, item, snapshot_price=item.product.price)
        product.stock -= item.qty
        product.sales += item.qty
        product.save()
    clear_cart(user)
```

### 8.4 支付创建与回调
```python
@transaction.atomic
create_payment(order_id):
    order = lock_order(order_id)
    if order.paid:
        return 400
    txn = get_or_create_pending_txn(order)
    qr = alipay.precreate(out_trade_no=txn.no, amount=order.total)
    txn.qr_code = qr
    txn.save()

notify_alipay(payload):
    if not verify_sign(payload):
        return "fail"
    txn = find_txn(payload.out_trade_no)

        mark_txn_paid(txn)
        mark_order_paid(txn.order)
    return "success"
```

## 11. 你了解 xxx 原理吗？（技术深挖）【已实现】

### 11.1 `select_for_update` 原理
用于行级锁，防止并发写冲突。必须在事务中使用，否则 Django 报事务管理错误。

### 11.2 JWT 原理
服务端签发 token，客户端携带 token，服务端验签与校验过期时间，实现无状态认证。

### 11.3 支付回调验签原理
第三方回调携带签名，服务端用平台公钥验签，验签通过后再更新订单状态。

### 11.4 Redis 验证码场景原理
验证码是短生命周期数据，适合 Redis 的高性能 key-value 存取与 TTL 失效机制。

### 11.5 Vite 代理原理
开发环境将前端 `/api` 请求转发到后端，规避浏览器跨域限制。
