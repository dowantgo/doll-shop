## 1. 介绍一下这个项目？【已实现】
### 场景背景
希望做一个包含“完整交易闭环”的项目，不是只做商品展示，而是覆盖从用户认证到支付完成的完整链路。

### 方案选择
采用前后端分离：前端负责交互体验，后端负责业务规则和数据一致性。

### 对比权衡
- 一体化模板开发速度快，但前后端职责耦合，不利于扩展。
- 分离架构联调成本更高，但更接近真实团队协作模式。

### 实现路径
- 前端：路由守卫 + API 分层 + 统一请求拦截。
- 后端：按业务域拆 app（users/products/cart/orders/payment）。
- 支付：创建支付单、轮询状态、异步回调验签。

### 效果指标
- 能完整演示核心电商链路。
- 能支持管理端运营动作（商品、订单、用户管理）。

### 风险边界
- 支付回调依赖公网可达地址。
- 高并发能力仍需二迭代强化。

### 可扩展点
二迭代接入评价、榜单、热榜轮播、性能优化和容器化。

## 6. xx 场景下，A 和 B 怎么选？【已实现 + 规划中】
### 场景样例 1：缓存 vs 数据库直查
- 读多写少热点（热榜、排行）优先缓存。
- 强一致写场景（扣库存、支付状态）必须以数据库事务为准。

### 场景样例 2：同步调用 vs 异步队列
- 用户即时反馈场景（下单响应）先同步最小闭环。
- 非核心链路（日志、推荐刷新）可异步队列化（迭代2/3规划）。

### 场景样例 3：乐观锁 vs 悲观锁
- 高冲突库存扣减优先悲观锁/原子扣减（当前已采用事务锁思路）。
- 冲突低可考虑乐观锁提升吞吐。

## 7. 项目难点是什么？【已实现】
### 难点 1：支付链路一致性
支付涉及第三方接口、回调验签、订单状态变更，容易出现“支付成功但订单未更新”。

### 难点 2：环境联调复杂
本地、ngrok、反向代理、支付回调等多环境切换容易出现配置漂移。

### 难点 3：事务边界
`select_for_update` 必须在事务中使用，否则会报错并引发 500。

### 我怎么解决
- 明确支付状态机与回调处理逻辑。
- 统一环境变量入口并规范启动手册。
- 在关键写路径上显式加事务。

## 11. 你了解 xxx 原理吗？（技术深挖）【已实现】

### 11.1 `select_for_update` 原理
用于行级锁，防止并发写冲突。必须在事务中使用，否则 Django 报事务管理错误。

### 11.2 JWT 原理
服务端签发 token，客户端携带 token，服务端验签与校验过期时间，实现无状态认证。

### 11.3 支付回调验签原理
第三方回调携带签名，服务端用平台公钥验签，验签通过后再更新订单状态。

### 11.4 Redis 验证码场景原理
验证码是短生命周期数据，适合 Redis 的高性能 key-value 存取与 TTL 失效机制。

### 11.5 Vite 代理原理
开发环境将前端 `/api` 请求转发到后端，规避浏览器跨域限制。

## 15. 管理员评价审核实现与接口契约【已实现】
### 场景背景
评价模块上线后，用户提交内容默认进入待审核状态。需要管理员进行审核，才能决定评价是否对外展示，避免违规内容直接透出。

### 方案选择
采用“用户写入待审 + 管理员审核流转”模式：
- 用户端 `POST /api/reviews/product/{id}/` 只写入 `pending`。
- 管理端 `PATCH /api/admin/reviews/{id}/audit/` 负责把状态变更为 `approved/rejected`。

### 对比权衡
- 方案 A（直出评价）：体验快，但内容风险高。
- 方案 B（全人工审核）：安全性高，但增加审核成本与延迟。
- 当前选择：默认待审 + 管理端审核，优先保证内容安全与可控。

### 实现路径
1. 评价数据模型 `Review` 增加审核字段：
   - `status`：`pending/approved/rejected`
   - `audit_remark`、`audit_by`、`audit_at`
2. 用户提交评价时写入 `pending`，并通过 `order_item` 一对一约束保证“一单项一评”。
3. 管理员审核接口校验角色权限（`request.user.role == 'admin'`）。
4. 审核接口仅允许写入：
   - `status`：`approved | rejected`
   - `audit_remark`：可选，最大 255 字符
5. 商品详情评价列表接口仅返回 `approved` 评价。

#### 接口契约（已实现）
- `PATCH /api/admin/reviews/{id}/audit/`
- 鉴权：`Bearer JWT` + 管理员角色
- 请求体：
```json
{
  "status": "approved",
  "audit_remark": "内容合规，审核通过"
}
```
- 成功响应（字段最小集）：
```json
{
  "id": 101,
  "product_id": 12,
  "user_id": 7,
  "rating": 5,
  "content": "手感很好，做工不错",
  "status": "approved",
  "created_at": "2026-04-16T10:00:00+08:00",
  "user_name": "user1",
  "product_name": "哈吉米",
  "audit_remark": "内容合规，审核通过"
}
```

### 效果指标
- 评价内容可控：默认不直出，审核后可见。
- 审核链路清晰：状态流转可追溯（审核人/审核时间/备注）。
- 业务一致性：商品详情只展示已通过评价。

### 风险边界
- 当前审核是单条人工操作，尚未提供批量审核能力。
- 暂未接入敏感词自动机与风控模型，仍依赖人工判断。
- 审核 SLA 未做系统级约束（超时提醒、分配队列等）。

### 可扩展点
- 当前已实现管理端评价待审列表与单条审核，后续可扩展批量审核能力。
- 加入敏感词预筛，降低人工审核负担。
- 增加审核日志表与操作审计检索。

### 流程级伪代码
```python
# user submit review
def create_review(user, product_id, rating, content):
    with transaction.atomic():
        order_item = lock_first_unreviewed_paid_order_item(user, product_id)
        if not order_item:
            return error("Only purchased products can be reviewed once")
        review = Review.create(
            order_item=order_item,
            user=user,
            product_id=product_id,
            rating=rating,
            content=content,
            status="pending",
        )
    return review


# admin audit review
def audit_review(admin_user, review_id, status, remark=""):
    assert admin_user.role == "admin"
    assert status in ["approved", "rejected"]
    review = Review.get(id=review_id)
    review.status = status
    review.audit_remark = remark
    review.audit_by = admin_user
    review.audit_at = now()
    review.save(update_fields=["status", "audit_remark", "audit_by", "audit_at", "updated_at"])
    return review


# product detail list
def product_reviews(product_id):
    return Review.filter(product_id=product_id, status="approved").order_by("-created_at")
```

## 16. 秒杀预埋骨架（活动模型、预占接口、幂等键、限流骨架）【预埋未上线】
### 场景背景
二迭代要为高并发交易做架构预埋，但本轮目标是“先预埋能力，不接入正式下单支付链路”，避免在功能迭代期引入交易风险。

### 方案选择
采用“预埋骨架 + 不上线交易”的策略：
- 先定义活动模型、预占接口契约、幂等键规范、限流骨架。
- 不直接修改当前线上订单支付主流程。
- 后续迭代再做完整库存扣减、异步下单、支付闭环。

### 对比权衡
- 直接上线秒杀交易流：价值高，但风险和联调复杂度极高。
- 只做文档方案：风险低，但工程落地准备不足。
- 当前选择：实现可对接的骨架契约，兼顾进度与风险控制。

## 17. 秒杀状态机与超时回收设计【规划中，迭代2补充项】
### 场景背景
秒杀场景中，用户可能预占成功但长时间不支付，导致库存被占用，影响其他用户购买与活动转化。

### 方案选择
采用“活动状态机 + 预占超时回收 + 订单超时联动回补”三层方案，确保库存及时释放并保持状态一致。

### 对比权衡
- 方案A：仅依赖订单超时关闭，不处理预占。
  - 优点：实现简单。
  - 缺点：预占库存可能长期占用，出现虚假售罄。
- 方案B：仅处理预占过期，不联动订单。
  - 优点：对秒杀模块改动小。
  - 缺点：已下单未支付场景仍可能占库存。
- 当前建议：方案C（双通道回收），同时处理预占过期和订单超时，保证最终一致性。

### 实现路径
1. 活动状态扩展：`draft/preheating/online/ended/offline`。
2. 预占超时任务：扫描 `reserved_expires_at <= now` 的 `reserved` 记录，回补库存。
3. 订单超时任务：扫描待支付超时订单，关闭订单并回补对应秒杀预占库存。
4. 状态迁移约束：禁止非法迁移（例如 `ended -> online`）。

### 效果指标
- 预占库存占用率下降。
- 超时订单处理时延稳定在可控窗口内。
- 活动库存显示与真实可售库存一致。

### 风险边界
- 定时任务并发执行可能重复回补，必须依赖行锁与幂等更新。
- 时区与时间源不统一会导致过期判定漂移。

### 流程级伪代码
```python
def expire_seckill_reservations(now):
    rows = query_reserved_expired(now)
    for r in rows:
        with transaction.atomic():
            lock(r)
            if r.status != "reserved":
                continue
            release_activity_reserved_stock(r.activity_id, r.qty)
            restore_product_stock(r.product_id, r.qty)
            r.status = "expired"
            save(r)


def close_timeout_orders(now):
    orders = query_pending_timeout_orders(now)
    for o in orders:

            lock(o)
            if o.payment_status != "pending":

                release_activity_reserved_stock(reservation.activity_id, reservation.qty)
                restore_product_stock(reservation.product_id, reservation.qty)
                reservation.status = "cancelled"
                save(reservation)
```

## 18. 支付回调补偿与最终一致性【规划中，迭代2补充项】
### 场景背景
第三方支付回调存在网络抖动、验签失败重试、服务瞬时不可用等情况，可能导致“用户已支付但平台订单仍待支付”。

### 方案选择
采用“回调实时处理 + 补偿任务兜底”的双通道一致性方案。

### 对比权衡
- 方案A：仅依赖实时回调。
  - 优点：实现成本低。
  - 缺点：回调丢失后无修复路径。
- 方案B：定时全量对账。
  - 优点：覆盖面大。
  - 缺点：开销高、实时性差。
- 当前建议：方案C（增量补偿），只扫描异常交易并重查支付平台状态。

### 实现路径
1. 记录支付交易状态与最后回调结果。
2. 补偿任务定时扫描“待支付超时但存在第三方交易号”的订单。
3. 调用平台查询接口确认真实交易状态。
4. 若为成功则补标订单支付成功；否则按策略重试或关闭。

### 效果指标
- “已支付未入账”问题可自动修复。
- 补偿成功率、平均修复时长可统计。
- 人工干预比例下降。

### 风险边界
- 补偿任务不应覆盖正常交易流，避免双写冲突。
- 查询频率需限速，防止触发第三方接口限流。

### 流程级伪代码
```python
def reconcile_pending_payments(now):
    txns = query_abnormal_pending_transactions(now)
    for txn in txns:
        with transaction.atomic():
            lock(txn)
            if txn.local_status in ["success", "closed"]:
                continue
            remote = query_gateway_status(txn.out_trade_no)
            if remote in ["TRADE_SUCCESS", "TRADE_FINISHED"]:
                mark_txn_success(txn)
                mark_order_paid(txn.order_id)
            elif should_close(txn, now):
                mark_txn_closed(txn)
                close_order_if_pending(txn.order_id)
            else:
                bump_retry_count(txn)
```

## 20. 评价敏感词策略（过滤、提示、误伤边界）【规划中，迭代2补充项】
### 场景背景
仅做“掩码过滤”虽然能防止违规词透出，但用户不知道为何被改写，体验不透明。

### 方案选择
采用“内容净化 + 命中反馈 + 可编辑建议”策略：
- 后端输出净化内容。
- 同时返回命中词信息（脱敏后）。
- 前端提示用户可修改后再提交。

### 对比权衡
- 仅过滤：实现简单，但用户困惑。
- 仅拦截：体验差，提交失败率高。
- 当前建议：过滤 + 提示并行，兼顾安全与体验。

1. 敏感词匹配服务返回命中列表与净化文本。
2. 评价提交响应新增字段：`hit_sensitive_words`, `sanitized_content`, `suggestion`。
3. 前端展示提示并允许用户一键替换/继续提交。

- 评价提交成功率提升。
- 用户投诉“莫名失败”比例下降。
- 敏感词误伤可追溯与持续优化。

### 风险边界
- 词库误伤（例如同音词、上下文无害词）需持续迭代。

```python
def sanitize_review_content(content):
```

### 13.2 超时回收为什么重要？怎么理解库存回补？
#### 问题
秒杀预占后如果用户一直不支付，为什么必须做超时回收？

#### 回答
秒杀预占本质上是在“抢库存”，如果不做超时回收，库存会被长时间占住，导致真实可售库存被虚占，最终出现活动看起来还有库存、实际却无法售卖的问题。超时回收的核心就是让预占单在 TTL 到期后自动失效，并把锁住的库存重新回补到可售库存里，这样才能保证系统既不超卖，也不长期浪费库存。

#### 伪代码索引
- 见第 8.3 节“下单事务与库存一致性”。
- 见第 9.2 节“预占接口”和第 9.5 节“幂等与并发”。

#### 业务落地结果
- 预占过期释放、库存回补已纳入当前文档基线。
- 测试时重点验证：到期后状态变化、库存恢复、并发下不超卖。

### 13.3 支付补偿为什么要单独抽出来？
#### 问题
支付成功后为什么还要做补偿？回调不是一次就够了吗？

#### 回答
支付回调是外部异步链路，天然存在网络抖动、回调超时、重复通知、签名验签失败等问题。单次回调不能保证系统最终一致，因此需要补偿机制兜底：当回调丢失或处理失败时，系统可以通过补偿任务重新查询第三方支付状态，把订单状态修正回来。这样才能避免“用户已经付款，但平台还显示未支付”的不一致问题。

### 13.4 敏感词反馈为什么不能只做“拦截”？

```python
item_paid_total = (order_item_total / order_subtotal) * order_final_payable
```

### 3) 支付补偿（手工 + 15分钟自动）【已实现】
#### 场景背景
支付回调天然不可靠，必须有补偿机制兜底，避免“已支付仍待支付”。

#### 方案选择
保留管理员手工补偿接口，同时引入 Celery Beat 每15分钟自动补偿任务。

#### 对比权衡
- 仅回调：链路短，但漏回调时不一致风险高。
- 回调 + 补偿：复杂度提高，但最终一致性更稳。

#### 实现路径
- 手工入口：`POST /api/pay/reconcile/`。
- 自动任务：`payment.reconcile_pending`，调度 `crontab(minute='*/15')`。
- 落审计字段：补偿次数、最后补偿时间、最后状态、最后错误。

#### 风险边界
- 自动补偿依赖 Celery worker/beat 正常运行。
- 第三方网关波动会导致补偿重试增加。

#### 伪代码
```python
for txn in pending_expired_transactions():
    remote = alipay_query(txn.out_trade_no)
    if remote.success:
        mark_paid(txn)
    elif remote.closed:
        mark_closed(txn)
    else:
        mark_retry(txn, remote.status)
```

### 4) 物流适配层（Mock / 快递100）【已实现】
#### 场景背景
物流查询需要支持本地可测和真实对接，不能把第三方实现硬编码在业务视图里。

#### 方案选择
引入 Provider 抽象：`MockProvider` 与 `Kuaidi100Provider`，由配置项 `LOGISTICS_PROVIDER` 切换。

#### 对比权衡
- 单一第三方直连：接入快但耦合重。
- 适配层：初期代码多一点，但后续扩展新渠道成本低。

#### 实现路径
- 物流查询服务：`apps/orders/services/logistics.py`。
- 订单轨迹接口：`GET /api/orders/orders/{id}/logistics/`。
- 失败降级：返回“暂无法获取轨迹”，不阻塞订单主流程。

#### 风险边界
- 快递100配置缺失或限流时只能走降级提示。
- 当前轨迹时效依赖三方接口可用性。

#### 伪代码
```python
provider = get_logistics_provider_from_settings()
result = provider.query(order)
return build_logistics_response(order, result)
```

### 5) 最终一致性（交易 + 履约）【部分已实现，部分规划中】
#### 场景背景
优惠、退款、支付补偿、物流并存后，系统目标从“单接口成功”升级为“跨域状态一致”。

#### 方案选择
以“状态机 + 幂等 + 补偿 + 审计”作为一致性主线，保证订单、支付、退款、履约可以追溯。

#### 对比权衡
- 强一致大事务：实现重、可用性风险高。
- 最终一致 + 补偿：符合支付与物流异步业务特性。

#### 实现路径
- 已实现：优惠快照、退款状态机、支付补偿任务、物流适配层。
- 规划中：真实网关退款回写、四账自动核对（订单/支付/退款/库存）、完善监控告警。

#### 风险边界
- 异步链路下需要接受短时不一致，但必须能收敛。
- 补偿策略要防止重复执行导致状态抖动。

#### 伪代码
```text
order_created -> coupon_locked -> payment_pending
payment_callback|reconcile_task -> order_paid
refund_request -> admin_review -> refunding -> refund_success|failed
shipping_update -> logistics_provider_query -> timeline_render
```
