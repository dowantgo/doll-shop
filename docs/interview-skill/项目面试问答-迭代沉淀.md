## 1. 介绍一下这个项目？【已实现】
### 场景背景
目标不是做一个只会展示商品的 CRUD 项目，而是做一个具有完整交易闭环的电商系统。

### 方案选择
采用前后端分离架构：前端负责体验与交互，后端负责规则、状态、一致性和接口契约。

### 对比权衡
- 一体化模板开发快，但前后端耦合重。
- 前后端分离联调成本更高，但更接近真实团队协作模式。

### 实现路径
- 前端：Vue3 + Vite + Element Plus
- 后端：Django + DRF + JWT
- 存储：MySQL + Redis
- 支付：支付宝沙箱 + Mock
- 异步补偿：Celery + Beat

### 效果指标
- 已具备完整交易闭环
- 支持后台管理与履约操作
- 支持性能治理与量化压测

## 4. 这个项目最大的难点是什么？【已实现】
### 难点 1：交易一致性
支付、优惠券、退款、库存这些模块都不是独立存在的，一个环节状态错了会影响整条链路。

### 难点 2：高并发场景下的秒杀一致性
要同时解决库存预占、限购、幂等、防超卖和超时释放问题。

### 难点 3：性能治理不是单点优化
Iter4 中发现，连一个日志探针都可能把核心接口拖慢，所以性能优化必须结合架构、缓存、查询、日志一起看。

## 5. 秒杀是怎么做的？【已实现】
### 方案
- 活动模型 + 商品维度秒杀配置
- 预占库存而不是直接成功下单
- 限购控制
- 幂等键防重复请求
- 订单关联秒杀预占状态

### 结果
- 在库存 20、100 请求 / 50 并发场景下，成功预占 20 单、失败 80 单，最终库存为 0，未超卖。
- 在 50 请求 / 25 并发下，同一用户同一幂等键仅生成 1 条预占记录。

### 伪代码
```python
if not allow_rate_limit(activity_id, user_id, ip):
    return reject()

if idem_key_exists(user_id, activity_id, idem_key):
    return old_result()

with transaction.atomic():
    lock_activity_stock()
    if available_stock <= 0:
        return sold_out()
    reserve_stock()
    create_reservation()
```

## 6. 优惠券和价格试算是怎么做的？【已实现】
### 方案
- 后端统一价格试算，不信任前端金额
- 规则固定：先满减，再单券抵扣
- 结果写入订单优惠快照，保证可复算

### 为什么这么做
如果把试算放在前端，金额口径会飘；如果不做快照，后续规则变更后订单金额无法追溯。

### 伪代码
```python
subtotal = calc_items_subtotal(items)
full_reduction = calc_full_reduction(subtotal)
base = max(subtotal - full_reduction, 0)
coupon_discount = calc_coupon_discount(user_coupon, base)
final_payable = max(base - coupon_discount, 0)
save_discount_snapshot(order, subtotal, full_reduction, coupon_discount, final_payable)
```

## 7. 部分退款为什么复杂？【已实现】
### 回答
因为退款不能按商品标价直接退，要按订单项在订单实付金额中的占比做分摊，否则会超退。

### 方案
- 按订单项实付比例计算可退金额
- 支持按数量退款
- 后台审核驱动状态机流转
- 幂等键防止重复提交

### 伪代码
```python
item_paid_total = (order_item_total / order_subtotal) * order_final_payable
unit_paid = item_paid_total / order_item_qty
requested = unit_paid * refund_qty
max_refundable = item_paid_total - already_refunded_amount(order_item)
approved_amount = min(requested, max_refundable)
```

## 8. 支付补偿为什么要做？【已实现】
### 回答
支付回调不是绝对可靠的，可能丢失、超时、验签失败，所以必须有补偿机制兜底，避免“用户已支付但订单仍待支付”。

### 方案
- 管理员手工补偿接口
- Celery Beat 每 15 分钟自动补偿
- 记录补偿次数、最后状态、最后错误、最后补偿时间

### 伪代码
```python
for txn in pending_expired_transactions():
    remote = alipay_query(txn.out_trade_no)
    if remote.success:
        mark_paid(txn)
    elif remote.closed:
        mark_closed(txn)
    else:
        mark_retry(txn, remote.status)
```

## 10. Iter4 最有代表性的优化案例是什么？【已实现 / 已验收】
### 回答
最典型的是 `price-preview` 的性能回退修复。

一开始为了增强日志，我在价格试算接口里加了一个 `payment_id` 探针。第一轮压测后发现，`price-preview` 的 P95 反而从 `716.53ms` 回退到了 `766.34ms`。后面排查发现，问题不是主业务逻辑，而是每次试算都会额外查询一次支付表。

我最后把它改成了“支付状态变更时写缓存，价格试算时只读缓存”，既保留了日志可观测性，又把热路径查库干掉了。修复后 `price-preview` 的 P95 降到了 `537.94ms`。

### 这个案例的价值
- 说明性能问题不一定来自业务本身，也可能来自探针、日志、附属逻辑
- 说明优化不能只盯 SQL，还要看整条请求链路

## 11. 为什么要做 service 分层？【已实现】
### 回答
因为如果把定价、优惠券释放、退款聚合、feed 缓存这些逻辑都堆在 view 里，后续不管是扩功能、压性能还是查 bug，成本都会越来越高。

我在 Iter4 里把这些规则拆到 service 层后，收益主要有三点：
1. 视图层只保留编排，职责更清晰
2. 性能问题更容易被单独定位
3. 面试和文档里也更容易讲清楚系统设计
