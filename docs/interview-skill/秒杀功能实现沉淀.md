# 秒杀功能实现沉淀（迭代2）

## 1. 为什么秒杀链路要“先预占，再下单”？
- 秒杀核心冲突是高并发抢库存，直接下单会放大数据库写压力。
- 预占阶段先做资格校验与库存锁定，把高峰请求收敛到轻量状态变更。
- 下单阶段再复用现有订单与支付体系，降低改造风险。

## 3. 关键一致性策略
- 幂等：`X-Idempotency-Key` 防重复预占。
- 并发控制：事务 + `select_for_update` 锁活动/预占记录。
- 状态机：`reserved -> ordered -> paid`，异常流转到 `cancelled/expired`。
- 反向补偿：订单取消/过期会释放预占并回补库存。

def pre_reserve(user, activity_id, quantity, idempotency_key):
    check_rate_limit(user)
    with transaction.atomic():
        activity = lock_activity(activity_id)
        product = lock_product(activity.product_id)
        assert activity.is_online_in_window()
        assert activity.remaining_stock >= quantity
        assert product.stock >= quantity
        reservation = create_reservation(user, activity, quantity, status='reserved')
        activity.reserved_stock += quantity
        product.stock -= quantity
        save(activity, product, reservation)
    return reservation
```

def create_order_from_reservation(user, reservation_id, address_id):
    with transaction.atomic():
        reservation = lock_reservation(user, reservation_id)
        assert reservation.status == 'reserved'
        assert reservation.not_expired()
        order = create_order(user, address_id, total=reservation.activity.seckill_price * reservation.quantity)
        create_order_item(order, reservation.product, reservation.quantity, reservation.activity.seckill_price)
        reservation.status = 'ordered'
        reservation.order = order
        reservation.reserved_expires_at = order.expires_at
        save(order, reservation)
    return order
```
