# 面试提问 Skill

## 目标
这份 Skill 用于把开发过程中的技术决策沉淀成可面试表达的资产，解决以下问题：
- 说不清项目价值、业务背景和个人贡献。
- 只能讲功能现象，讲不出技术取舍与底层原理。
- 面试追问时没有结构化回答路径。

目标不是“背答案”，而是形成一套可复用的表达框架：
- 业务价值可讲清。
- 技术方案可对比。
- 实现细节可落地。
- 风险边界可说明。
- 后续扩展可展开。

## 输入规范
每次复盘一个功能或模块时，输入至少包含：
- 功能名称：例如“支付创建与回调”。
- 业务场景：用户在什么流程下触发它。
- 角色与职责：你负责了什么，哪些是他人负责。
- 技术栈上下文：前端、后端、数据库、缓存、中间件。
- 关键代码位置：核心文件、核心函数、核心接口。
- 结果数据：性能指标、稳定性结果、线上反馈或测试结论。

建议附带：
- 1 段核心伪代码。
- 1 个失败案例和修复过程。
- 1 个替代方案对比结论。

## 输出规范
每个面试题统一输出结构：
1. 场景背景：业务目标和约束。
2. 方案选择：为什么选当前方案。
3. 对比权衡：与替代方案的优劣和边界。
4. 实现路径：核心流程、核心接口、关键数据结构。
5. 效果指标：性能、正确性、稳定性、可维护性。
6. 风险边界：哪些场景可能失效，怎么兜底。
7. 可扩展点：下一迭代怎么扩展。

回答标准：
- 优先真实，不夸大。
- 已实现与规划中必须明确区分。
- 尽量用“业务 + 技术 + 数据”三段式收束。

## 提问分层

### L1 项目全局层
- 介绍一下这个项目。
- 为什么做这个项目，业务背景是什么。
- 你在项目中负责什么。
- 这个项目最有价值的地方是什么。

### L2 方案决策层
- 为什么做某个功能（例如支付、订单、验证码）。
- 为什么选某技术栈（例如 Django/DRF、Vue3、MySQL、Redis）。
- 你有考虑过替代方案吗，和方案 A/B 比较如何。
- 某场景下为什么选 X 不选 Y。

### L3 实现落地层
- 某功能具体怎么实现。
- 核心接口和数据模型是什么。
- 状态机、事务、并发一致性怎么保证。
- 错误处理和兜底策略是什么。

### L4 性能与稳定性层
- 性能优化做了哪些。
- 如何定位性能瓶颈。
- 高并发场景如何防超卖、限流、幂等。
- 如何做监控与告警。

### L5 原理深挖层
- JWT、缓存、数据库索引、事务隔离、支付验签原理。
- 前端请求拦截、状态恢复、路由守卫原理。
- 反向代理与内网穿透链路原理。

### L6 项目延伸层
- 如果重做一次，你会怎么改。
- 未来迭代（秒杀、评价、榜单、容器化）怎么规划。
- 如何做工程化（CI/CD、自动化测试、环境隔离）。

## 追问模板

### 模板 1：技术选型追问
- 主问：为什么选择这个技术？
- 追问 A：替代方案是什么？为什么没选？
- 追问 B：在什么场景下你会换方案？
- 追问 C：这个方案的最大风险是什么？

### 模板 2：实现细节追问
- 主问：具体是怎么实现的？
- 追问 A：核心流程可否用伪代码说明？
- 追问 B：失败场景如何处理？
- 追问 C：怎么验证它真的生效？

### 模板 3：性能优化追问
- 主问：你做了哪些性能优化？
- 追问 A：优化前后指标对比如何？
- 追问 B：怎么定位到这个瓶颈？
- 追问 C：有没有副作用，如何规避？

### 模板 4：难点与复盘追问
- 主问：项目难点是什么？
- 追问 A：为什么难？
- 追问 B：你怎么拆解并解决？
- 追问 C：如果再做一次会如何设计？

## 禁忌项
- 禁止把“规划中”说成“已上线”。
- 禁止只讲框架名，不讲业务约束与权衡。
- 禁止只讲 happy path，不讲异常和边界。
- 禁止只讲做了什么，不讲为什么这样做。
- 禁止堆砌术语而没有代码级落地。

## 每次需求完成后的沉淀动作（强制）
每个需求必须补齐三类信息：
1. 为什么做：业务背景 + 目标收益。
2. 怎么做：实现流程 + 关键伪代码。
3. 取舍与复盘：替代方案 + 风险边界 + 下一步。

执行规则：
- 需求开发完成当天沉淀。
- 没有伪代码的记录不算完成。
- 标注实现状态：已实现 / 规划中。

## 使用方式
- 开发前：按“提问分层”做技术设计自问，提前发现薄弱点。
- 开发中：按“输出规范”记录关键决策和证据。
- 开发后：按“复盘模板”沉淀到项目问答总文档。
