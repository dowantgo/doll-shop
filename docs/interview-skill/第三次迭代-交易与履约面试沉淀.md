## 1. 优惠引擎（满减 + 单券同享）【已实现】
### 场景背景
Iter3 目标是把“优惠”纳入交易主链路，而不是仅在前端做展示计算。要求同一订单支持“平台满减 + 一张用户券”共同生效，并且结果可追溯、可复算。

### 方案选择
采用后端统一结算策略：
1. 先算商品小计 `subtotal`。
2. 再算满减 `full_reduction`。
3. 最后算单券抵扣 `coupon_discount`。
4. 落库 `OrderDiscountSnapshot`，保存定价版本和快照。

### 对比权衡
- 方案A：前端试算为主，后端只信任传参。开发快，但风险高，容易被篡改。
- 方案B：后端统一计算并落快照。开发成本高一点，但金额口径统一、审计能力强。
- 最终选择方案B，优先保证交易一致性。

### 实现路径
1. 新增优惠域模型：`CouponTemplate`、`UserCoupon`、`OrderDiscountSnapshot`。
2. 订单接口新增试算能力：`POST /api/orders/orders/price-preview/`。
3. 订单接口新增锁券能力：`POST /api/orders/orders/{order_id}/apply-coupon/`。
4. 订单创建时同步写入优惠快照，支付成功后将锁定券标记为已使用。

### 风险边界
- 当前规则固定“最多1张券 + 满减同享”，不支持多券叠加。
- 券状态流转依赖订单状态，需要严格保证取消/支付时的回滚与置位。

### 伪代码
```python
def calc_order_final(subtotal, user_coupon):
    full_reduction = calc_full_reduction(subtotal)
    base = max(subtotal - full_reduction, 0)

    coupon_discount = 0
    if user_coupon:
        ok, msg = validate_coupon_for_amount(user_coupon, base)
        if not ok:
            raise BizError(msg)
        coupon_discount = calc_coupon_discount(user_coupon, base)

    final = max(base - coupon_discount, 0)
    return {
        "subtotal": subtotal,
        "full_reduction": full_reduction,
        "coupon_discount": coupon_discount,
        "final": final,
    }
```

## 2. 部分退款金额分摊（按商品实付）【已实现】
### 场景背景
用户购买多件商品且使用了优惠后，退款不能直接按原价退；否则会造成“超退”或“少退”。Iter3 采用“按订单项实付占比分摊”的退款模型。

### 方案选择
基于订单实付金额（优先取 `discount_snapshot.final_payable_amount`）进行订单项分摊，再按退款数量计算本次应退金额。

### 对比权衡
- 方案A：按商品标价退款。实现简单，但与实际支付不一致。
- 方案B：按实付分摊退款。规则更复杂，但财务口径正确。
- 最终选择方案B。

### 实现路径
1. 新增退款域：`RefundRequest` + `RefundAuditLog`。
2. 用户发起：`POST /api/refunds/`，支持 `idempotency_key` 防重。
3. 校验流程：
   - 仅已支付订单可申请；
   - 按订单项剩余可退数量限制；
   - 按已退金额累计限制，避免超退。
4. 后台审核：`PATCH /api/admin/refunds/{id}/review/`，状态机：
   `pending -> approved/rejected -> refunding -> success/failed`。

### 风险边界
- 当前边界明确“不含运费分摊”。
- 同一订单项多次退款的并发竞争，需要依赖数据库聚合校验和幂等键。

### 伪代码
```python
def calc_item_refund_amount(order, order_item, req_qty):
    subtotal = sum(i.price * i.quantity for i in order.items)
    final_payable = order.discount_snapshot.final_payable_amount if order.discount_snapshot else order.total_price

    item_base = order_item.price * order_item.quantity
    item_paid_total = round((item_base / subtotal) * final_payable, 2)

    unit_paid = round(item_paid_total / order_item.quantity, 2)
    requested = round(unit_paid * req_qty, 2)

    already_refunded = sum_valid_refund_amount(order_item)
    refundable = max(item_paid_total - already_refunded, 0)
    return min(requested, refundable)
```

## 3. 支付补偿（手工 + 15分钟自动）【已实现】
### 场景背景
支付回调是异步链路，存在“用户已付款，但平台状态未更新”的情况。Iter3 要求补偿机制可手动触发，也可周期任务自动收敛。

### 方案选择
双通道补偿：
1. 手工补偿：管理员接口 `POST /api/pay/reconcile/`。
2. 自动补偿：Celery Beat 每15分钟执行任务 `payment.reconcile_pending`。

### 对比权衡
- 方案A：仅依赖回调。实现最省，但漏回调时会长期不一致。
- 方案B：回调 + 补偿闭环。复杂度上升，但可靠性显著提升。
- 最终选择方案B。

### 实现路径
1. `PaymentTransaction` 增加补偿审计字段：
   `reconcile_attempts`、`last_reconcile_at`、`last_reconcile_status`、`last_reconcile_error`。
2. 补偿逻辑按交易状态分支处理：
   - `TRADE_SUCCESS/TRADE_FINISHED` -> 标记已支付；
   - `TRADE_CLOSED` -> 关闭本地交易；
   - 其他状态 -> 记录重试信息。
3. 使用行锁 + 幂等判断避免重复补偿造成状态抖动。

### 风险边界
- 自动补偿依赖 Celery Worker + Beat 同时在线。
- 第三方接口波动会增加重试次数，需要监控失败率与错误分布。

### 伪代码
```python
@shared_task(name="payment.reconcile_pending")
def reconcile_pending():
    for txn in pending_expired_alipay_transactions():
        remote = query_alipay(txn.out_trade_no)
        if remote.trade_status in ["TRADE_SUCCESS", "TRADE_FINISHED"]:
            mark_paid_idempotent(txn, remote.trade_no)
        elif remote.trade_status == "TRADE_CLOSED":
            close_txn_idempotent(txn)
        else:
            record_retry(txn, f"trade_status={remote.trade_status}")
```

## 4. 物流适配层（Mock / 快递100）【已实现】
### 场景背景
物流查询不应把第三方 SDK 直接写死在业务视图里，否则后续切供应商会牵动大量代码。Iter3 引入 Provider 适配层统一抽象。

### 方案选择
抽象 `BaseLogisticsProvider`，实现：
- `MockProvider`：本地可测、无外部依赖。
- `Kuaidi100Provider`：真实三方链路。
通过环境变量 `LOGISTICS_PROVIDER` 动态切换。

### 对比权衡
- 方案A：业务层直调单一第三方。接入快，但耦合高、替换成本大。
- 方案B：适配层隔离。首期代码稍多，但扩展成本低、测试更稳定。
- 最终选择方案B。

### 实现路径
1. 在订单域新增物流服务入口：`apps/orders/services/logistics.py`。
2. 订单详情接口新增轨迹查询动作：`GET /api/orders/orders/{id}/logistics/`。
3. 未发货/未支付/三方失败场景统一降级返回，保证前台可解释。

### 风险边界
- 快递100配置缺失时只能返回降级结果。
- 第三方限流或异常时，前台只能展示“暂无法获取轨迹”。

### 伪代码
```python
def get_logistics(order):
    provider = get_provider_from_env()  # mock or kuaidi100
    result = provider.query(order)
    return {
        "order_id": order.order_id,
        "provider": result.provider,
        "available": result.available,
        "message": result.message,
        "traces": result.traces,
    }
```

## 5. 最终一致性（交易 + 履约）【部分已实现，部分规划中】
### 场景背景
Iter3 同时引入优惠、退款、补偿、物流后，系统关注点从“单接口成功”升级到“跨域状态一致”。

### 方案选择
以“状态机 + 幂等 + 审计日志”为主线，确保订单、支付单、退款单、物流状态能互相对齐。

### 对比权衡
- 强一致全同步事务：实现困难且影响可用性。
- 最终一致 + 补偿机制：工程可落地，符合支付/物流异步特性。

### 实现路径
1. 支付侧采用“回调 + 手工补偿 + 自动补偿”收敛支付状态。
2. 退款侧采用状态机流转并记录审计日志。
3. 订单侧保留优惠快照，确保退款与对账可追溯。
4. 物流侧采用查询降级，不阻塞订单主流程。

### 风险边界
- `已实现`：状态机、幂等键、补偿任务、审计字段、适配层框架。
- `规划中`：
  - 自动退款到真实支付网关（当前为同步模拟路径）；
  - 四账自动核对报表（订单/支付/退款/库存）；
  - 分布式链路追踪与告警看板。

### 伪代码
```text
create_order -> lock_coupon -> pay_pending
pay_callback or reconcile_task -> order_paid
user_refund_request -> admin_review -> refunding -> refund_success/failed
shipping_update -> logistics_query(provider) -> timeline_view
```
