# 第五个迭代-5.2测试计划

## 1. 测试背景

Iter5.2 对秒杀链路做专项稳定性改造，核心变化不是新增页面功能，而是把秒杀预占阶段从“先打数据库”调整为：

- 入口限流
- 一次性 submit token
- Redis 原子库存预占
- Redis 幂等结果回放
- reservation token 驱动的建单
- Celery 异步清理与回补

因此，本轮测试重点不是普通页面点击，而是验证：

- 高并发下是否仍然满足不超卖、不重复成功下单、失败可回补。
- Redis 预占、判重、submit token、reservation token、补偿回补是否形成完整闭环。
- 新方案是否实质性减少秒杀预占阶段对数据库的直接冲击。

## 2. 测试目标

### 2.1 功能目标

- 验证秒杀主链路在正常场景下可成功预占、建单、支付、完成。
- 验证异常场景下能明确失败，且不会破坏库存与订单状态。

### 2.2 一致性目标

- 验证库存预占、用户判重、重复建单、重复补偿都保持幂等。
- 验证 Redis 与 MySQL 最终状态可对齐、可解释。

### 2.3 性能目标

- 验证流量在入口和 Redis 层被前置削峰后，数据库不再承担秒杀预占阶段的主要热点竞争。
- 验证售罄和重复请求场景能快速失败，减少无效写库。

## 3. 测试范围

### 3.1 本轮覆盖

- 秒杀入口限流
- 秒杀 submit token
- Redis 库存预占
- Redis 判重与幂等回放
- `reservation_token` 建单
- 过期票据异步清理
- 订单取消/订单过期后的库存回补
- 后台活动变更与 Redis 同步
- 并发压测与指标采集

### 3.2 本轮不覆盖

- 普通下单链路专项复测
- 非秒杀支付通道专项改造
- 新消息中间件接入测试
- 真正的 MQ 异步建单队列

## 4. 测试策略

### 4.1 分层策略

- 单元/组件级：验证 Redis 票据、submit token、幂等回放、回补函数。
- 接口级：验证状态码、业务语义、结果收敛。
- 集成级：验证 Redis、Celery、MySQL 的状态流转。
- 并发级：验证秒杀高峰下的吞吐、拒绝模式和库存一致性。
- 回归级：验证本轮改造不破坏既有订单、支付、后台管理基础链路。

### 4.2 结果判断原则

- 优先看一致性是否成立，其次看热点压力是否下降。
- 若性能未达预期但一致性成立，结论应归类为“可用但待优化”，不能误判为功能失败。
- 若出现超卖、重复成功建单、重复回补，则直接判定为 P0 阻塞。

## 5. 测试环境与前置条件

### 5.1 环境要求

- Django 后端服务可正常启动。
- MySQL、Redis、Celery Worker、Celery Beat 均可用。
- 测试环境使用独立 Redis 库，避免与其他测试数据混用。
- 时钟保持一致，避免 token 过期和活动时间窗误判。

### 5.2 数据准备

至少准备以下数据：

- 进行中的秒杀活动 1 个，库存 20。
- 进行中的秒杀活动 1 个，库存 1，用于极限竞争。
- 未开始活动 1 个。
- 已结束活动 1 个。
- 可登录普通用户账号至少 5 个。
- 压测用批量账号若干。

### 5.3 观测准备

测试前确认可采集以下数据：

- Redis 库存桶键和值
- submit token 使用状态
- reservation token 票据状态
- 幂等结果映射状态
- 过期清理任务执行日志
- MySQL 秒杀订单与 reservation 记录
- 数据库连接数、接口耗时、限流命中数等指标

## 6. 核心测试点

### 6.1 入口限流

#### 正向

- 合法用户在频率范围内可正常发起秒杀。
- 活动开始后首次请求可进入 Redis 预占逻辑。

#### 反向

- 单 IP 高频请求命中限流并快速失败。
- 单用户短时间重复点击命中限流并快速失败。
- 未登录用户被拦截。
- 未开始、已结束、已下架活动被快速拒绝。

#### 重点观察

- 限流命中后是否绕过 Redis 预占。
- 是否记录限流日志和指标。

### 6.2 submit token

#### 正向

- 合法 token 可成功消费一次。
- 获取 token 后可正常发起预占。

#### 反向

- token 重复使用被拒绝。
- 过期 token 被拒绝。
- 伪造 token 被拒绝。

#### 重点观察

- token 失败是否不消耗库存。
- token 消费后是否确实失效。

### 6.3 Redis 库存预占

#### 正向

- 库存充足时预占成功，Redis 库存减 1。
- 库存为 1 时并发多个请求，仅 1 个成功。

#### 反向

- 库存为 0 时请求快速失败。
- 售罄后继续请求不应进入数据库建单主路径。
- 异常中断后库存不能出现负数。

#### 重点观察

- Redis 预占是否原子。
- 最终成功数是否严格小于等于初始库存。

### 6.4 Redis 判重与幂等回放

#### 正向

- 相同用户、相同 `X-Idempotency-Key` 的重复请求，可回放同一张 `reservation_token`。
- 已有数据库 reservation/order 时，重复请求可回放已有结果。

#### 反向

- 不同幂等键不会错误复用同一票据。
- 幂等结果过期后不应造成脏回放。

#### 重点观察

- 判重是否早于重复扣库存。
- Redis 回放与数据库回放是否语义一致。

### 6.5 `reservation_token` 建单

#### 正向

- Redis 预占成功后可用 `reservation_token` 创建订单。
- 建单成功后会物化 `SeckillReservation`、`Order`、`OrderItem`。
- 重复建单请求会返回已有订单结果。

#### 反向

- 失效 ticket 无法建单。
- 已过期 reservation 无法建单。
- 活动下线或库存状态异常时建单会安全失败并回补。

#### 重点观察

- 同一 `reservation_token` 最终只生成一张订单。
- 建单成功后 Redis ticket 是否被清理。

### 6.6 过期清理与补偿回补

#### 正向

- Redis ticket 过期后，Celery 任务可回补 Redis 库存。
- 秒杀订单超时未支付后，库存与 reservation 状态可收口。
- 人工取消订单后状态与库存可收口。

#### 反向

- 重复执行补偿不会重复加库存。
- 已支付订单不允许误回补库存。
- 不存在的 ticket/已释放 ticket 再次处理时应安全幂等。

#### 重点观察

- 回补后 Redis 和 MySQL 是否对齐。
- `reason` 指标和日志是否可追踪。

### 6.7 后台活动变更与 Redis 同步

- 后台创建活动后 Redis 库存桶被初始化。
- 后台调库存、改状态后 Redis 库存桶同步更新。
- 删除活动后 Redis 库存桶被清理。

## 7. 并发与性能测试计划

### 7.1 基线场景

- 20 并发抢 20 库存。
- 50 并发抢 20 库存。
- 100 并发抢 20 库存。
- 200 并发抢 20 库存。
- 100 并发抢 1 库存。

### 7.2 重点观测指标

- 请求总数
- 预占成功数
- 限流命中数
- submit token 失败数
- 幂等回放命中数
- Redis 库存不足失败数
- `reservation_token` 建单成功数
- ticket 过期释放数
- 订单取消/过期回补数
- 数据库连接数峰值
- 秒杀主接口 P95 / P99 响应时间

### 7.3 验证重点

- 成功数不能超过库存数。
- 库存售罄后失败应以前置拒绝和 Redis 快速失败为主。
- 数据库连接峰值应显著低于旧的“同步预占写库”路径。
- 错误模式应以受控业务失败为主，而不是大量数据库连接异常。

## 8. 用例清单

### 8.1 P0 用例

- `TC-I5.2-SK-001`：活动开始后合法用户可成功预占秒杀请求。
- `TC-I5.2-SK-002`：库存为 1 时并发请求仅 1 个成功。
- `TC-I5.2-SK-003`：同一幂等键重复提交仅回放同一 ticket，不重复扣库存。
- `TC-I5.2-SK-004`：submit token 仅可使用一次。
- `TC-I5.2-SK-005`：同一 `reservation_token` 只成功建单一次。
- `TC-I5.2-SK-006`：ticket 过期后库存可补偿回补。
- `TC-I5.2-SK-007`：未支付超时关闭后库存可回补且不重复回补。
- `TC-I5.2-SK-008`：高并发场景下最终成功数不超过初始库存。

### 8.2 P1 用例

- `TC-I5.2-SK-009`：单 IP 高频请求命中入口限流。
- `TC-I5.2-SK-010`：单用户快速重复点击命中限流。
- `TC-I5.2-SK-011`：过期 token 被拒绝且不扣库存。
- `TC-I5.2-SK-012`：售罄后请求快速失败且不物化数据库 reservation。
- `TC-I5.2-SK-013`：重复建单请求返回已有订单结果。
- `TC-I5.2-SK-014`：后台调库存和状态后 Redis 同步正确。

### 8.3 P2 用例

- `TC-I5.2-SK-015`：未开始活动请求快速失败。
- `TC-I5.2-SK-016`：已结束活动请求快速失败。
- `TC-I5.2-SK-017`：非法 token 或伪造 token 被拒绝。
- `TC-I5.2-SK-018`：重复补偿命中时记录幂等日志。

## 9. 通过准则

### 9.1 功能通过

- P0 用例全部通过。
- P1 用例无阻塞级缺陷。
- 秒杀主流程、失败流程、补偿流程均有明确结果。

### 9.2 一致性通过

- 无超卖。
- 无重复成功建单。
- 无重复回补库存。
- Redis 与 MySQL 结果可核对。

### 9.3 性能通过

- 同口径压测下，秒杀预占阶段的数据库压力明显下降。
- 库存售罄后绝大多数请求能在入口或 Redis 层快速结束。
- 秒杀主接口的错误收敛模式可稳定复测，不再由旧的同步预占写库主导失败模式。

## 10. 风险与关注项

- 如果 Redis 预热库存与数据库活动库存初始值不一致，测试结论可能失真。
- 如果异步清理任务日志不完整，会影响 ticket 过期和补偿路径判断。
- 如果测试环境与旧方案环境差异过大，性能对比只能做趋势判断，不能做绝对结论。
- 如果后续继续演进为 MQ 异步建单，则本轮测试结论不能直接替代下一阶段验收。

Iter5.2 测试的核心不是只看“抢没抢到”，而是验证秒杀链路是否真正具备 Redis 前置削峰、票据化预占、幂等回放和失败回补后的高并发稳定性。
