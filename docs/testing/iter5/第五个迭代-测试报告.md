# 第五个迭代测试报告

- 版本：Iter5
- 测试对象：并发治理与交易稳定性增强
- 依据文档：`docs/requirements/iter5/第五个迭代需求.md`

## 1. 测试范围

本轮重点覆盖：

- 用户入口限流
- Refresh Token 完整续签
- 购物车原子更新
- 普通下单幂等
- 订单过期异步清理
- 支付轮询减压
- Iter5.2 的秒杀 Redis 化与前置削峰改造

## 2. 当前阶段结论

Iter5 到目前为止已经完成两段主要工作：

1. **Iter5.1**
   - 入口限流、Refresh Token、购物车原子更新、普通下单幂等、过期订单异步化、支付轮询减压
2. **Iter5.2**
   - 秒杀提交 token
   - Redis 预占库存桶
   - Redis 判重与结果回放
   - `reservation_token` 下单收口
   - 过期预占异步回补

## 3. 测试执行摘要

| 指标 | 说明 |
| --- | --- |
| 功能结论 | 功能闭环通过 |
| 中等压力结论 | 明显改善 |
| 300 并发结论 | 尚未完全收口 |
| 主要瓶颈 | MySQL `1040 Too many connections` |

## 4. 已验证结论

1. 登录限流、验证码限流、Refresh Token 链路方向正确。
2. 购物车原子更新与普通下单幂等有效。
3. 订单过期异步清理与支付轮询减压方向正确。
4. 秒杀 Redis 化链路已验证通过：
   - token 一次性消费
   - 同 `X-Idempotency-Key` 结果复用
   - `reservation_token` 下单收口
   - 过期预占异步回补

## 5. 当前主要问题

1. 高压下最先暴露出的不是业务逻辑错误，而是 MySQL 连接数瓶颈。
2. `price-preview` 已通过入口护栏明显减压。
3. `create_from_cart` 在 100 并发下已改善，但 300 并发下仍受数据库连接上限约束。
4. 支付 `query/status` 缓存优先命中有效，但在更高并发下仍会被数据库连接瓶颈牵连。

## 6. 测试结论

第五个迭代当前可定义为：

- 功能验收：通过
- 架构方向验证：通过
- 中等压力验收：通过
- 高压容量验收：未完全通过

因此 Iter5 可以继续推进，但应明确区分：
- **Iter5.1 / 5.2 已完成阶段性目标**
- **后续阶段仍需继续围绕高压稳定性与异步化深化**
