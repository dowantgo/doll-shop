# 第五个迭代测试结论与整改清单

## 一、当前测试结论

### 1. 功能结论
- Iter5 本轮功能闭环通过。
- 登录限流、验证码限流、Refresh Token 续签、购物车原子更新、普通下单幂等、过期订单异步清理、支付轮询减压等核心能力方向正确。
- 交易主链路没有出现需要推翻重做的业务错误。

### 2. 一致性结论
- 普通下单幂等成立。
- 低库存场景防超卖成立。
- 中等压力下，支付轮询减压与状态收敛逻辑有效。
- 当前暴露的主要问题不是交易模型错误，而是高压下基础设施瓶颈先出现。

### 3. 高压结论
- 在 300+ 并发场景下，MySQL 连接数开始大量触顶，出现 `1040 Too many connections`。
- 1000 并发场景下，该问题进一步放大。
- 这会连带影响：
  - `price-preview`
  - `create_from_cart`
  - 购物车高压测试完成量
  - 支付查询高压路径

### 4. 定性结论
- Iter5 第一版已经完成“功能增强版”目标。
- Iter5 当前未完成“高压稳定版”目标。
- 下一步不需要推翻交易模型，而应进入第二轮稳态优化。

## 二、开发整改优先级

## P0：本轮必须处理

### 1. MySQL 连接数瓶颈治理
当前最先暴露出来的是数据库连接资源耗尽，而不是业务正确性问题。

#### 本轮处理方向
- 核对并调整 MySQL `max_connections`
- 检查线程缓存和连接复用参数
- 优化 Django 连接复用配置
- 复测 100 / 300 并发，确认不再过早被连接数打满

#### 应用层已收口方向
- 增加连接复用配置
- 减少高频重复请求直接冲击 MySQL

### 2. 普通下单 Redis 第一层幂等挡板
MySQL `OrderSubmission` 作为最终幂等事实记录继续保留，但在其之前增加 Redis 第一层处理态挡板，降低重复请求直接打库的概率。

#### 本轮处理方向
- 下单前增加 Redis `SETNX` 幂等锁
- 命中后快速返回“处理中/已处理”
- 成功请求缓存首单结果，减少重复请求再次走数据库

### 3. 重接口入口保护
高压下优先保护最容易成为热点的两个入口：

- `price-preview`
- `create_from_cart`

#### 本轮处理方向
- 增加单用户限流
- 增加单 IP 限流
- 对 `price-preview` 增加资源粒度限流
- 命中后快速失败，优先保护数据库

## P1：建议继续补强

### 1. 支付轮询减压继续优化
当前 Redis 缓存和限流已经生效，但高压时仍受数据库连接数影响。

#### 后续建议
- 提高 `status/query` 缓存优先命中率
- 单独统计轮询限流命中
- 单独统计第三方查询次数
- 补一轮高压下缓存命中率验证

### 2. 购物车高压边界复测
当前购物车高压不足以直接定性为业务丢更新，因为数据库连接瓶颈已经先暴露。

#### 后续建议
- 连接数治理完成后，单独复测 100 / 300 并发加购
- 验证最终数量是否严格等于成功请求数

## P2：后续可选方向

### 1. 秒杀链路进一步 Redis 化
如果后续要继续强化高并发能力，可优先把秒杀链路升级为：
- Redis 库存桶
- Redis 判重
- 队列削峰
- 异步落库
- 失败补偿

### 2. Refresh Token 安全补强
当前登录、刷新、登出链路已可用，后续可以继续增加：
- Refresh Token 黑名单
- 服务端主动吊销

## 三、测试侧给开发的最终判断

- 功能闭环是通过的。
- 中等压力下，一致性和减压能力是成立的。
- 当前主要问题不是业务逻辑错误，而是 **MySQL 连接数瓶颈先暴露**。
- 幂等下单和低库存防超卖逻辑都成立，开发不需要推翻交易模型。
- 现阶段最划算的动作是：**连接数治理 + Redis 前置削峰/幂等 + 重接口入口保护**。

第二轮整改的目标不是扩业务，而是把 Iter5 从“功能增强版”继续推进到“高压稳定版”。

本轮整改完成后，重点要重新验证：
- 100 / 300 并发下 MySQL 连接是否仍然先触顶
- `create_from_cart` 重复请求是否更少冲击数据库
- `price-preview` 高频访问是否能更早被挡在入口
- 支付查询高压路径是否更多命中缓存
