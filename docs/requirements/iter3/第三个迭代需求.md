# 第三个迭代需求（Iter3）

- 文档版本：v3.1
- 更新时间：2026-04-20
- 适用分支：`iter3/main`
- 文档状态：已按当前代码落地情况同步

## 1. 迭代目标

在 Iter2（秒杀、评价、榜单）基础上，补齐交易闭环与履约能力：
1. 优惠与秒杀可同享（同订单引擎下统一定价）。
2. 支持部分退款（按订单项+数量）。
3. 增强支付一致性（15 分钟自动补偿 + 手工补偿）。
4. 引入物流适配层（Mock + 快递100）。

## 2. 范围定义

## 2.1 本轮上线（P0）

### 2.1.1 优惠券与订单试算
1. 新增券模板、用户券、订单优惠快照。
2. 试算规则：先满减，再单券。
3. 单订单最多一张券。
4. 支持待支付订单应用券并锁券。

### 2.1.2 部分退款
1. 用户按订单项和数量发起退款。
2. 退款金额按订单项实付分摊。
3. 后台审核同意/拒绝并记录审计日志。
4. 幂等键防重复申请。

### 2.1.3 支付补偿
1. 保留手工补偿接口：`POST /api/pay/reconcile/`
2. 新增自动补偿任务：每 15 分钟执行一次。
3. 支付单记录补偿尝试信息（次数/状态/错误/时间）。

### 2.1.4 物流轨迹
1. 新增物流 provider 抽象。
2. 支持 `mock` 和 `kuaidi100` 两类 provider。
3. 前台订单详情可查看轨迹时间线。
4. 第三方失败时降级返回提示，不阻断主流程。

## 2.2 本轮不阻塞发布（P1/P2）
1. 多券组合与最优券自动选择。
2. 第三方真实退款回执全链路打通。
3. 多物流商并行策略与熔断。

## 4. 核心业务规则

## 4.1 优惠规则
1. 满减与券可叠加，顺序固定“满减 -> 单券”。
2. 订单优惠快照必须落库。
3. 订单取消/超时时，已锁券回退为未使用。

## 4.2 退款规则
1. 仅已支付订单可申请退款。
2. 不允许超量退款与重复并发越界退款。
3. 状态机：
   - `pending -> approved/rejected -> refunding -> success/failed`

## 4.3 对账补偿规则
1. 自动任务与手工补偿均应幂等。
2. 补偿仅收敛状态，不重复记账。
3. 补偿日志必须可追踪到订单号与支付单号。

## 4.4 物流规则
1. 未发货返回空轨迹。
2. Provider 错误返回降级提示。
3. 业务层不直接耦合第三方 SDK。
