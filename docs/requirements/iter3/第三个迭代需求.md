# 第三个迭代需求

## 1. 迭代目标
在第二次迭代（评价、榜单、秒杀基础能力）之上，第三次迭代聚焦“交易闭环稳定性 + 运营增长能力 + 履约可视化”。

## 2. 范围与优先级

### 2.1 P0（必须完成）

#### 2.1.1 优惠券与满减引擎
- 业务目标：提升转化率和客单价。
- 核心功能：
  - 优惠券模板（满减/立减/折扣）
  - 用户领取与发放
  - 下单结算时校验（有效期、门槛、叠加规则）
  - 订单落库保存优惠明细
- 验收标准：
  - 同一订单优惠计算结果可复现（前后端一致）
  - 非法券不可用并返回明确原因

#### 2.1.2 售后退款模块
- 业务目标：形成完整售后闭环。
- 核心功能：
  - 申请退款（仅已支付可申请）
  - 管理员审核（同意/拒绝）
  - 退款状态流转与记录
  - 与支付模块联动（原路退回/模拟退款）
- 验收标准：
  - 退款状态机完整、可追溯
  - 重复退款、并发退款被拦截

#### 2.1.3 支付对账与补偿任务
- 业务目标：避免“支付成功但订单未更新”。
- 核心功能：
  - 定时对账任务
  - 回调失败补偿重试
  - 异常订单告警与修复入口
- 验收标准：
  - 异常单可自动收敛到最终一致状态
  - 补偿日志可追踪（订单号、重试次数、结果）

#### 2.1.4 物流轨迹与发货工作台
- 业务目标：降低客服压力，提升履约体验。
- 核心功能：
  - 后台发货录入（快递公司/单号）
  - 前台订单详情查看物流轨迹
  - 签收确认与异常状态展示
- 验收标准：
  - 已发货订单可查看最新物流状态
  - 未发货不展示物流轨迹

### 2.2 P1（建议完成）

#### 2.2.1 活动中心统一化
- 统一管理秒杀、直降、满减活动。
- 活动模板化，支持并行活动与冲突规则。

#### 2.2.2 搜索增强
- 搜索联想词、热搜词、排序策略（销量/价格/上新）。
- 支持关键词埋点统计。

#### 2.2.3 用户资产体系
- 收藏、浏览足迹、积分、等级权益。
- 为后续会员体系做预埋。

#### 2.2.4 库存中心增强
- 库存流水、库存预警、锁定库存与回补日志。
- 降低超卖与脏库存风险。

### 2.3 P2（可延期）

#### 2.3.1 消息通知中心
- 站内信/邮件模板化通知（下单、支付、发货、退款）。

#### 2.3.2 运营数据看板
- GMV、转化率、复购率、活动 ROI、热销趋势。

## 3. 子代理拆分建议（开发组织）

### 3.1 可解耦并行
- BE-交易：优惠券、退款、对账补偿
- BE-履约：物流轨迹、发货工作台、库存流水
- FE-商城：结算页优惠展示、订单物流、售后入口
- FE-后台：活动中心、退款审核、物流录入
- QA/Doc：测试用例、回归基线、接口契约校验

### 3.2 不可解耦（主代理统筹）
- 订单金额计算一致性（优惠/退款/支付）
- 支付结果与订单状态最终一致性
- 跨模块事务边界与补偿策略

## 4. 技术与质量门禁
- 功能门禁：P0 全部通过冒烟与回归。
- 一致性门禁：订单、支付、退款状态一致。
- 性能门禁：关键接口 P95 可观测且有基线。
- 可观测性门禁：异常日志可定位到订单/用户/请求链路。

## 5. 交付物清单
- 需求文档（本文件）
- 接口契约更新
- 数据库迁移脚本
- 回归测试用例与测试报告
- 面试沉淀补充（交易一致性、补偿机制、活动系统设计）

## 6. 迭代边界
- 本迭代默认不做跨境支付、复杂仓配、多商户结算。
- 优先保证单体电商核心链路可稳定复用。
