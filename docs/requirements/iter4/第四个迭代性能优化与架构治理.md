# 第四个迭代需求：性能优化与架构治理

## 1. 迭代定位

第四个迭代不新增大型业务功能，重点面向第三个迭代已经完成的交易闭环做专项治理：

- 缓存稳定性：统一商品榜单和热榜缓存策略，补齐命中/未命中日志。
- 查询效率：为订单、支付、退款、秒杀、商品 feed 等高频查询补充索引。
- 架构分层：将定价、优惠券释放、商品 feed 缓存等逻辑从视图层下沉到 service 层。
- 可观测性：继续保留 `order_id`、`payment_id`、`refund_no` 等关键日志字段，便于测试和定位。
- 测试资产：以第三迭代性能数据为基线，产出 Iter3 vs Iter4 对比报告。

## 3. 本轮优化目标

| 模块 | 优化目标 | 验收口径 |
|---|---|---|
| 商品 feed | 统一缓存 key、版本号、TTL、命中率统计、失效入口 | `top-sales` / `hot-feed` P95 目标降至 300ms 内 |
| 价格试算 | 定价逻辑下沉 service，减少视图层复杂度 | `price-preview` P95 目标低于 450ms |
| 退款创建 | 合并退款数量/金额聚合查询，补充退款高频索引 | `refund-create` P95 目标低于 400ms |
| 物流查询 | 保持适配层稳定，减少不必要 ORM 查询 | `logistics-query` P95 保持 400ms 内 |
| 数据库索引 | 补充 feed、订单、支付、退款、秒杀扫描索引 | 迁移可执行，查询路径与业务字段匹配 |
| 架构治理 | 视图层只保留编排，复杂规则下沉 service | 接口路径、参数、响应字段不漂移 |

## 4. 已落地改动范围

### 4.1 商品 feed 缓存治理

- 新增 `ProductFeedService`，集中处理 `top-sales`、`hot-feed` 的缓存读写、DB 回源和日志。
- 缓存 key 继续沿用 `products:feed:{feed}:v{version}:limit:{limit}` 结构。
- 新增命中/未命中计数：`products:feed:stats:{feed}:hit`、`products:feed:stats:{feed}:miss`。
- 商品新增、编辑、删除、库存调整、图片变更等仍通过 `bump_feed_version_on_commit()` 统一失效。
- `ProductFeedSerializer` 改为使用已预取的图片集合，避免每个商品主图再触发额外查询。

### 4.2 交易 service 分层

- 新增 `orders.services.pricing`：
  - `build_pricing_preview()`：统一满减、优惠券、最终应付金额计算。
  - `pending_payment_probe()`：为性能和日志探针提供轻量支付单号读取。
- 新增 `orders.services.lifecycle`：
  - `release_locked_coupon()`：统一订单取消、过期清理、重新用券场景下的锁券释放逻辑。

### 4.3 退款查询优化

- `validate_refund_request()` 查询订单时补充 `select_related('discount_snapshot')`，避免退款金额分摊时额外查询优惠快照。
- 新增 `order_item_refund_usage()`，一次聚合得到已退款数量和已退款金额，替代原先两次聚合。

### 4.4 数据库索引治理

新增索引覆盖以下高频路径：

- 商品 feed：`status + sales + created_at`、`status + is_hot + hot_sort_order`
- 订单列表/过期扫描：`user + payment_status + status + created_at`、`payment_status + status + expires_at`
- 物流查询：`tracking_no`
- 支付补偿扫描：`payment_method + status + expire_time`
- 退款创建/幂等：`order_item + status`、`user + idempotency_key`
- 秒杀列表/过期释放：`group_id + status + is_enabled`、`status + reserved_expires_at`

## 5. 不变更范围

- 不新增用户侧业务 API。
- 不修改现有接口路径。
- 不修改前端调用字段。
- 不改变优惠、秒杀、退款、支付补偿的业务规则。
- 不把 `abc.lock`、本地缓存、虚拟环境、运行日志纳入提交。

## 7. 面试沉淀方向

- Redis feed 缓存：版本号失效为什么比逐 key 删除更稳。
- 查询优化：索引设计如何围绕过滤条件和排序字段建立。
- N+1 治理：为什么序列化主图时不能直接使用 `obj.images.filter()`。
- Service 分层：为什么把定价、优惠券释放、退款聚合逻辑从视图层拆出来。
- 简历量化：只写复测后真实下降幅度，不写未验证的优化值。
