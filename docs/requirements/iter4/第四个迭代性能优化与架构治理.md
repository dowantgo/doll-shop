# 第四个迭代需求：性能优化与架构治理

## 1. 迭代定位

第四个迭代不新增大型业务功能，重点围绕第三个迭代已经完成的交易闭环做专项治理，目标是在不改变业务口径和接口契约的前提下，进一步提升性能、稳定性和可维护性。

本轮治理重点：
- 缓存稳定性：统一商品热榜、销量排行缓存策略，补齐命中率统计与失效入口。
- 查询效率：围绕订单、支付、退款、秒杀、商品 feed 等高频路径补充索引与 ORM 预取优化。
- 架构分层：将定价、优惠券释放、退款聚合、商品 feed 缓存等复杂逻辑从视图层下沉到 service 层。
- 可观测性：继续保留 `order_id`、`payment_id`、`refund_no` 等关键日志字段，并降低日志探针对主链路性能的影响。
- 测试资产：以 Iter3 真实性能数据作为基线，输出 Iter3 vs Iter4 对比报告。

## 3. 已落地优化项

### 3.1 商品 feed 缓存治理
- 新增 `ProductFeedService`，统一处理 `top-sales` 与 `hot-feed` 的缓存读写、DB 回源、TTL 和统计逻辑。
- 统一缓存 key 结构，延续版本号失效方案。
- 新增命中/未命中计数键，用于计算真实命中率。
- 商品新增、编辑、删除、库存/销量变化、图片变更等场景统一触发失效。
- 主图序列化复用预取结果，降低 N+1 风险。

### 3.2 交易 service 分层
- 新增 `orders.services.pricing`，统一定价与价格试算逻辑。
- 新增 `orders.services.lifecycle`，统一优惠券释放与订单生命周期治理。

### 3.3 退款与查询优化
- 退款创建链路合并已退款数量和金额聚合查询。
- 订单、支付、退款、秒杀、商品高频查询补齐索引。

### 3.4 Price Preview 回退修复
第一轮 Iter4 压测中，`price-preview` 的 P95 从 Iter3 的 `716.53ms` 回退到 `766.34ms`。定位后发现，问题主要来自新增的 `payment_id` 日志探针：每次价格试算都会额外查询一次最近的待支付交易单。

修复方案：
- 将 `payment_id` 探针从“每次试算查支付表”改为“支付创建/支付成功/关闭/过期时维护缓存，试算时直接读缓存”。
- 保留日志可观测性，不丢 `payment_id` 字段。
- 增加支付探针缓存生命周期测试。

对应修复提交：`5df94a8`
