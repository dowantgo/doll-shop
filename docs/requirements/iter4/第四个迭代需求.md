# 第四个迭代需求

## 1. 迭代定位

第四个迭代不再扩大型业务功能，重点围绕第三个迭代已经完成的交易闭环做专项治理，目标是在不改变业务口径和接口契约的前提下，提升性能、稳定性和可维护性。

## 2. 迭代目标

- 统一商品热榜和销量排行缓存策略
- 优化订单、支付、退款、秒杀、商品 feed 等高频查询
- 将复杂交易逻辑从视图层下沉到 service 层
- 保留并优化关键链路日志探针
- 基于 Iter3 基线产出 Iter4 性能优化结果

## 3. 需求范围

### 3.1 缓存治理

- `top-sales` 和 `hot-feed` 缓存统一
- 补齐缓存命中率统计
- 统一缓存失效入口
- 商品变更时同步失效相关 feed 缓存

### 3.2 查询优化

- 高频查询补充数据库索引
- 通过 `select_related` / `prefetch_related` 降低 N+1 查询
- 优化退款聚合和物流查询链路

### 3.3 架构分层

- 抽离订单定价 service
- 抽离订单生命周期治理 service
- 抽离商品 feed service

### 3.4 可观测性

- 保留 `order_id`、`payment_id`、`refund_no` 等关键日志字段
- 降低日志探针对主链路的性能影响
- 保留性能压测与结果对比能力

## 4. 关键结果

- `top-sales` P95 从 `566.02ms` 降至 `328.51ms`
- `hot-feed` P95 从 `511.33ms` 降至 `272.78ms`
- `refund-create` P95 从 `557.44ms` 降至 `441.51ms`
- `logistics-query` P95 从 `366.26ms` 降至 `348.76ms`
- `price-preview` P95 从 `716.53ms` 降至 `537.94ms`

## 5. 验收标准

- 功能回归通过
- 性能对比无回退项
- 关键缓存命中率达到预期
- 文档、README、架构说明同步更新
