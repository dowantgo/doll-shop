# 第五个迭代需求-5.2：秒杀链路 Redis 化与前置削峰改造

## 1. 背景

Iter5.1 已经完成入口限流、购物车原子更新、普通下单幂等、订单过期异步清理、支付轮询减压等稳定性增强，但测试也明确暴露出一个现实问题：

- 在高压场景下，MySQL 连接资源会先成为瓶颈。
- 秒杀链路虽然已有活动、预占、状态流转和回补能力，但核心热点请求仍然过早依赖数据库。

因此 Iter5.2 不再扩业务，而是专门围绕秒杀主链路做第二阶段优化：

- 把无效流量挡在入口或 Redis 之前。
- 把库存竞争从数据库尽量前移到 Redis。
- 给秒杀请求增加一次性 token 和 Redis 判重，降低重复点击和重放带来的打库压力。
- 用 Redis 预占票据替代“每次预占都先落数据库”的旧路径。
- 保留当前同步建单闭环，同时把过期释放和回补收口到异步任务里。

> 说明：本轮 **不引入 MQ，也不落地完整的异步建单队列**。这部分属于后续可继续演进的方向，但不作为 Iter5.2 的本次交付范围。

## 2. 本轮目标

### 2.1 业务目标

- 秒杀链路在高并发下保持“不超卖、不重复成功下单、失败可回补”的核心约束。
- 用户在秒杀开始瞬间得到更快、更稳定的结果反馈，避免大量无效请求直接冲击数据库。
- 库存售罄、重复请求、非法 token 等场景能够快速失败。

### 2.2 技术目标

- 将秒杀库存竞争主路径切换为 Redis 原子预占。
- 建立秒杀一次性 submit token 机制。
- 建立 Redis 用户判重和幂等结果回放。
- 将秒杀预占从“先写 MySQL”改为“先写 Redis 票据，创建订单时再物化数据库记录”。
- 将过期释放、订单取消、订单超时后的库存回补统一收口到补偿逻辑中。

### 2.3 非目标

- 不改造普通下单链路为同一套秒杀架构。
- 不引入 Kafka、RocketMQ 等新的消息中间件。
- 不把 Redis 作为交易主账，交易最终事实仍以 MySQL 为准。
- 不做复杂分库分表和跨机房容灾设计。

## 3. 本轮范围

本轮落地范围固定为：

- 秒杀入口限流
- 秒杀一次性 submit token
- Redis 库存桶预占
- Redis 用户判重与幂等结果回放
- reservation token 驱动的秒杀建单
- 秒杀活动后台操作同步 Redis 库存桶
- 过期票据释放、订单取消/超时回补
- 结构化指标与日志补齐

## 4. 现状问题

### 4.1 入口保护仍然偏弱

如果秒杀接口没有足够前置的限流和快速失败，活动开始瞬间的大量重复点击、脚本请求、库存已售罄后的无效请求，都会继续消耗应用和数据库资源。

### 4.2 秒杀预占仍过早打到数据库

旧路径里，预占成功意味着较早写入数据库记录。这样虽然正确性可控，但会把热点压力更快传到 MySQL。

### 4.3 判重和提交防重不够前置

同一用户短时间内重复点击、刷新页面重试、网络抖动重放请求，都会制造无效竞争。即使最终不超卖，也会放大热点资源消耗。

### 4.4 清理与补偿仍需统一收口

秒杀过期释放、订单取消、订单超时未支付，本质上都是库存和状态回收问题。如果入口链路自己“顺手做清理”，就会让职责混乱且不利于高压控制。

## 5. 方案总览

本轮 5.2 实际落地链路为：

1. 用户在秒杀下单前先获取一次性 `submit_token`。
2. 秒杀提交时，先做入口限流、活动可用性校验、submit token 校验。
3. 通过 Redis 原子逻辑完成：库存预占、票据写入、幂等结果缓存。
4. 预占成功后，接口直接返回 `reservation_token`，此时 **不立即写数据库预占记录**。
5. 用户继续发起“创建秒杀订单”请求时，再通过 `reservation_token` 物化 `SeckillReservation + Order + OrderItem`。
6. 订单创建成功后清理 Redis 预占票据；如果后续订单取消、过期未支付，则通过补偿逻辑回补 Redis 库存和数据库状态。
7. 过期未消费的 Redis 票据由 Celery 定时任务异步清理。

## 6. 详细需求

### 6.1 秒杀入口限流

#### 目标

在请求真正进入 Redis 预占前，先过滤高频、恶意和无效流量。

#### 本轮要求

- 秒杀预占接口增加用户级、IP 级、活动级限流。
- 活动未开始、已结束、已下线、商品已下架等场景快速失败。
- 限流命中写入结构化日志和指标。

#### 验收结果

- 限流命中后不进入库存预占逻辑。
- 测试可明确区分：用户限流、IP 限流、活动热点保护。

### 6.2 一次性 submit token

#### 目标

把“是否有资格发起一次秒杀提交”前移成一次性令牌校验，减少重放和重复点单。

#### 本轮要求

- 用户在发起秒杀前通过接口获取 submit token。
- token 绑定用户和活动，设置有效期。
- token 仅可消费一次，消费后立即失效。
- token 过期、伪造、重复使用时直接拒绝。

#### 验收结果

- submit token 可正常签发和消费。
- token 重复使用会失败，且不会扣减库存。

### 6.3 Redis 库存预占与票据化预留

#### 目标

把预占阶段从“先写 MySQL”切换为“先在 Redis 预占库存并生成票据”。

#### 本轮要求

- 活动库存可同步到 Redis 库存桶。
- 秒杀预占时通过 Redis 原子逻辑完成：
  - 校验库存是否足够
  - 扣减库存
  - 写入 reservation ticket
  - 写入幂等结果映射
- Redis 预占成功后返回 `reservation_token`。
- Redis 预占失败时直接返回库存不足或活动不可用，不进入数据库建单主路径。

#### 验收结果

- Redis 库存不得出现负数。
- 同一库存 under high concurrency 成功数不得超过初始值。
- 预占成功后数据库中可以暂时没有 `SeckillReservation` 记录。

### 6.4 Redis 判重与幂等回放

#### 目标

在重复请求到达业务核心前快速识别“这次其实已经抢过了/已经受理了”。

#### 本轮要求

- 秒杀预占继续接受 `X-Idempotency-Key`。
- 若相同用户和相同幂等键已在 Redis 中存在结果映射，则直接回放同一张 `reservation_token`。
- 若幂等结果已落到数据库，也支持回放已有预占或订单结果。
- 重复创建订单阶段增加基于 `reservation_token` 的处理锁，避免并发建单。

#### 验收结果

- 同一幂等键重复请求可以稳定返回同一结果。
- 同一 `reservation_token` 不会重复成功建单。

### 6.5 reservation token 驱动的秒杀建单

#### 目标

将数据库写入延后到真正“确认创建秒杀订单”这一步，减少预占阶段的打库压力。

#### 本轮要求

- `create-order` 接口支持使用 `reservation_token` 建单。
- 如果数据库还没有 `SeckillReservation`，则在建单时物化：
  - `SeckillReservation`
  - `Order`
  - `OrderItem`
- 物化成功后：
  - 更新商品销量
  - 更新活动 `reserved_stock`
  - 清理 Redis 票据
- 若数据库中已有同 token 的 reservation/order，则重复请求直接返回已有订单结果。

#### 验收结果

- 同一个 `reservation_token` 最终只会落一张订单。
- 建单后数据库与 Redis 状态一致。

### 6.6 过期清理与补偿回补

#### 目标

把 Redis 票据过期、订单取消、订单超时后的库存与状态回收统一收口。

#### 本轮要求

- Redis 票据过期后，由 Celery 定时任务异步扫描并回补库存。
- 订单取消、订单超时未支付时，同步回补 Redis 库存桶，并收口数据库 reservation 状态。
- 回补逻辑必须幂等，重复执行不能重复加库存。

#### 验收结果

- 票据过期后库存能恢复。
- 订单取消或过期后库存能恢复。
- 重复清理、重复补偿不会造成二次回补。

### 6.7 后台活动变更同步 Redis

#### 目标

避免后台改了秒杀活动库存或状态，但 Redis 仍保留旧值，导致运行时出现脏状态。

#### 本轮要求

- 后台创建活动时同步初始化库存桶。
- 后台调库存、改状态、批量创建时同步库存桶。
- 删除活动时清理 Redis 对应库存桶。

#### 验收结果

- 后台变更后，前台读到的剩余库存与 Redis 一致。

## 7. Redis 键与状态建议

### 7.1 关键键

- `iter5.2:seckill:submit-token:{user}:{activity}:{token}`
- `iter5.2:seckill:stock:{activity}`
- `iter5.2:seckill:reservation:{reservation_token}`
- `iter5.2:seckill:result:{user}:{idempotency_key}`
- `iter5.2:seckill:create-order:{reservation_token}`
- `iter5.2:seckill:reservation-expire-index`

### 7.2 状态语义

- `reserved`：Redis 预占成功或数据库 reservation 已预占。
- `ordered`：订单已创建，待支付。
- `paid`：支付成功。
- `cancelled`：订单取消后已回补。
- `expired`：超时过期后已回补。

## 8. 可观测性要求

至少记录以下指标：

- 秒杀 submit token 签发数
- submit token 失效或重复消费数
- 秒杀入口限流命中数
- Redis 预占成功数
- Redis 库存不足失败数
- 秒杀建单处理锁命中数
- Redis 票据过期释放数
- 订单取消/超时导致的回补数
- 幂等回放命中数

日志至少应包含：

- `activity_id`
- `product_id`
- `user_id`
- `idempotency_key`
- `reservation_token`
- `reason`

## 10. 验收标准

### 10.1 功能验收

- 秒杀预占前具备入口限流和 submit token 校验。
- 秒杀库存以 Redis 原子预占为主。
- 同一用户同一幂等键可稳定回放。
- 同一 `reservation_token` 不会重复成功建单。
- 票据过期和订单取消/过期后可以完成库存回补与状态收口。

### 10.2 一致性验收

- 任意并发场景下不得出现超卖。
- 重复请求、重复建单、重复补偿都不造成重复订单或重复加库存。
- Redis 与 MySQL 最终账务可核对、可解释。

### 10.3 性能验收

- 秒杀预占阶段的数据库写压力明显低于旧同步预占路径。
- 库存售罄后，大部分请求可在入口或 Redis 层快速失败。
- 秒杀主接口在同口径压测下，不再以数据库连接打满作为首要失败方式。

### 10.4 文档验收

- Iter5.2 需求、测试计划、测试结论口径一致。
- Redis 键设计、回补规则、验收标准有明确记录。

## 11. 一句话总结

Iter5.2 的核心是把秒杀主链路前移到 Redis：入口先限流，预占先发 token，库存先占坑，建单再物化，失败再回补，从而把真正的数据库热点挡在后面。
