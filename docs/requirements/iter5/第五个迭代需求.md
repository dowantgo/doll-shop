# 第五个迭代需求：并发治理与交易稳定性增强

## 1. 迭代背景

前四个迭代已经完成商城核心业务闭环，覆盖：

- 用户认证
- 商品管理
- 购物车
- 订单
- 支付
- 秒杀
- 优惠券
- 部分退款
- 物流查询
- 后台管理
- 性能优化与架构治理

当前项目已经具备较完整的业务能力，但在入口安全、并发治理、交易幂等、支付状态查询、任务异步化和认证续签方面仍有明显提升空间。

第五个迭代不再新增大型业务模块，重点转向：

- 用户入口防刷
- 购物车并发更新
- 普通下单请求幂等
- 订单过期异步清理
- 支付轮询减压
- Refresh Token 完整续签
- 任务治理与统一指标采集

本轮目标是把项目从“功能比较全”进一步推进到“稳定性和工程成熟度更高”。

## 2. 当前遗留问题

### 2.1 用户认证入口防刷能力不足

当前已具备图片验证码、邮箱验证码和 JWT 登录，但仍缺少系统性保护：

- 登录接口没有失败锁定
- 图片验证码接口没有访问频率限制
- 邮箱验证码发送没有冷却时间与频率限制
- 缺少按 IP、邮箱、用户名维度的限流

### 2.2 购物车并发更新不是原子操作

当前购物车加购和数量修改在并发下仍可能出现：

- 同一用户重复加购导致数量丢失更新
- 数量修改没有实时库存校验
- 重复点击造成购物车状态错乱

### 2.3 普通下单链路缺少请求级幂等

当前普通下单已经具备事务控制和库存扣减一致性，但缺少：

- 提交幂等键
- 重复请求收敛
- 网络重试 / 重复点击防重

### 2.4 订单过期清理仍夹在查询链路中

当前订单列表查询仍会顺带执行过期订单清理逻辑，存在：

- 读接口夹带写操作
- 高并发下接口职责混杂
- 查询性能和状态修复逻辑耦合

### 2.5 支付状态轮询成本偏高

当前支付链路已经有支付补偿，但支付前台流程仍然主要依赖轮询，导致：

- 支付状态查询接口热点明显
- 数据库读压力偏高
- 第三方状态查询成本偏高

### 2.6 Refresh Token 链路未完整走通

当前底层使用 SimpleJWT，但前端主要只持有 access token，缺少：

- refresh token 实际使用流程
- access token 过期自动续签
- 更安全的 refresh token 存放方式
- 退出登录后的 refresh token 失效清理

### 2.7 可观测性还不够系统

当前已有结构化日志、压测和命中率统计，但还缺少统一指标口径，例如：

- 限流命中数
- 幂等命中数
- 任务修复数
- 支付轮询减压效果

## 3. 本轮目标

第五个迭代的目标分为四层：

1. 提升入口安全和抗刷能力
2. 提升交易链路并发稳定性
3. 降低支付和补偿链路资源浪费
4. 补齐 JWT 双 Token 和统一指标基础

## 4. 需求范围

### 4.1 P0：本轮必须完成

#### 需求 1：登录 / 验证码 / 邮箱验证码接口限流

目标：

- 登录接口按 `IP + username/email` 维度记录失败次数
- 10 分钟内失败 5 次后锁定 15 分钟
- 图片验证码接口按 IP 限制为 20 次 / 分钟
- 邮箱验证码按邮箱 60 秒冷却、30 分钟最多 5 次；按 IP 10 次 / 10 分钟
- 所有限流命中记录结构化日志和 Redis 计数

#### 需求 2：购物车并发更新改为原子操作

目标：

- 对已存在购物车项使用 `F('quantity') + n`
- 并发加购场景避免丢失更新
- 修改数量时补库存校验
- 数量更新失败时返回统一错误
- 补充并发测试

#### 需求 3：普通下单链路增加请求级幂等

目标：

- 普通下单接口统一接收请求头 `X-Idempotency-Key`
- 服务端持久化订单提交记录，唯一键为 `user + idempotency_key`
- 重复请求直接返回首次结果
- 前端下单按钮增加提交态保护

#### 需求 4：过期订单清理彻底异步化

目标：

- 订单列表接口不再顺手执行 `_cleanup_expired_orders()`
- 由 Celery Beat 周期触发过期订单扫描任务
- 清理任务负责订单取消、库存回补、销量修正、优惠券释放、秒杀预占回收
- 增加重试与结构化日志

#### 需求 5：支付状态轮询减压优化

目标：

- 前端轮询策略退避：`2s * 3 -> 5s * 6 -> 10s`
- 服务端对 `pending` 支付状态增加短时缓存
- 支付状态查询接口增加限频
- 第三方支付状态查询优先走本地状态和缓存

### 4.2 核心 P1：本轮建议完成

#### 需求 6：Refresh Token 完整续签链路补齐

目标：

- 登录接口返回 access token，并写入 refresh token HttpOnly Cookie
- 增加 refresh 接口，自动轮换 refresh token Cookie
- 前端请求层在 access token 过期时自动续签并重试原请求
- 登出时清空 refresh token Cookie

#### 需求 7：补偿任务体系统一治理

目标：

- 统一任务命名和重试策略
- 为订单过期清理和支付补偿增加 summary 日志
- 为后续死信和告警预留结构化字段

#### 需求 8：统一性能与异常指标采集

目标：

- 关键接口统一记录耗时日志
- 补充错误数、限流命中数、幂等命中数、任务修复数、支付补偿成功数
- 将 Redis 缓存命中率纳入 Iter5 测试报告

### 4.3 P2：本轮不做

以下内容不纳入本轮实现：

- 热点商品库存去锁化重构
- 外部监控系统（Prometheus / Grafana）
- 全量 refresh token 黑名单体系
- 死信队列与复杂任务调度平台

## 5. 技术改造方向

### 5.1 用户入口治理

- Redis 计数器
- 登录失败锁定
- 验证码发送冷却与频率限制
- 结构化日志 + 限流指标

### 5.2 购物车并发治理

- `F()` 原子更新
- 并发安全的 get/create/update 流程
- 实时库存校验

### 5.3 下单幂等治理

- 前端 `X-Idempotency-Key`
- 服务端提交记录持久化
- 重复请求命中直接返回首次订单结果

### 5.4 异步任务治理

- Celery + Beat 统一承接订单过期清理
- 支付补偿和订单清理统一日志口径
- 标准化任务重试和失败记录

### 5.5 认证完善

- access / refresh 分离
- refresh token 走 HttpOnly Cookie
- 自动续签与退出登录清理

### 5.6 可观测性增强

- 关键接口耗时日志
- 限流 / 幂等 / 补偿 / 缓存命中计数
- 形成 Iter5 对比与稳定性报告基础
