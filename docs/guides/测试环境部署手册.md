# 测试环境部署手册

适用场景：

- 测试人员从 GitHub 仓库拉取指定迭代版本进行测试
- 需要快速启动前后端、初始化数据、执行回归测试
- 需要尽量减少与开发本地环境互相污染

## 1. 推荐交付方式

测试人员不要直接拉开发中的 `main` 进行测试，建议优先使用：

1. 稳定测试分支，例如 `release/iter1-test`
2. 或开发明确指定的 commit
3. 测试通过后，再由开发创建 Git Tag，例如 `v1.0.0-iter1`

推荐开发交付内容：

1. 指定测试分支名或 commit id
2. `backend/.env.test.example`
3. 初始化数据脚本
4. 测试账号
5. 已知问题清单
6. 当前迭代需求文档和测试用例

## 2. 测试人员本地准备

建议环境：

- Python 3.10+
- Node.js 18+
- MySQL 8+
- Redis 6+

建议使用独立测试资源：

- MySQL 测试库：`shop_test`
- Redis 测试库：`redis://127.0.0.1:6379/1`

这样做的目的，是避免把测试数据写进开发数据库或开发缓存。

## 3. 从 GitHub 拉取测试版本

如果开发给的是测试分支，使用：

```powershell
git clone <repo-url>
cd shop
git checkout release/iter1-test
```

如果开发给的是指定 commit，使用：

```powershell
git clone <repo-url>
cd shop
git checkout <commit-id>
```

如果本轮测试已经通过，且 GitHub 上已经创建 tag，才使用：

```powershell
git checkout v1.0.0-iter1
```

## 4. 第一次启动前需要做什么

### 4.1 准备数据库和 Redis

确保本机已经启动：

- MySQL
- Redis

并提前创建测试库：

```sql
CREATE DATABASE shop_test DEFAULT CHARACTER SET utf8mb4;
```

### 4.2 准备后端虚拟环境

```powershell
cd backend
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
```

### 4.3 准备测试环境变量

仓库已提供：

- [backend/.env.test.example](d:/shop/backend/.env.test.example)

复制为：

```powershell
copy .env.test.example .env
```

按实际环境修改：

- `DB_NAME`
- `DB_USER`
- `DB_PASSWORD`
- `REDIS_URL`

如果本轮只做功能回归，不做真实支付宝联调，可以先保留：

- `MOCK_PAYMENT_ENABLED=True`

## 5. 推荐启动方式

### 方式 A：一键启动

从仓库根目录执行：

```powershell
scripts\start_test_env.bat
```

这个脚本会做这些事：

1. 检查后端虚拟环境
2. 检查并生成 `backend\.env`
3. 检查前端依赖
4. 执行 Django migrate
5. 初始化测试数据
6. 分别启动后端和前端

启动后访问：

- 前端：`http://127.0.0.1:5173`
- 后端：`http://127.0.0.1:8000`

### 方式 B：手工启动

后端：

```powershell
cd backend
venv\Scripts\activate
python manage.py migrate
python init_data_runner.py
python manage.py runserver 0.0.0.0:8000
```

前端：

```powershell
cd frontend
npm install
npm run dev -- --host 0.0.0.0 --port 5173
```

## 6. 测试账号

初始化脚本会生成：

- 管理员：`admin / admin123`
- 普通用户：`user1 / user123`

如果初始化失败，请优先检查：

1. MySQL 是否启动
2. Redis 是否启动
3. `backend\.env` 是否正确

## 7. 测试数据重置

当你一轮测试做完，想恢复到“可重复执行”的初始状态时，执行：

```powershell
scripts\reset_test_data.bat
```

这个脚本会执行：

1. `flush`
2. `migrate`
3. `init_data_runner.py`

适合：

- 重跑同一批测试用例
- 回归测试前恢复数据基线
- 避免旧订单、旧购物车、旧地址影响结论

## 8. 支付测试建议

### 8.1 日常回归

优先使用 mock 支付。

原因：

- 不依赖真实支付宝配置
- 不依赖公网回调
- 更适合高频回归

### 8.2 提测验收

如果需要验证真实支付宝支付回调，需要：

1. 配置真实的支付宝参数
2. 将 `ALIPAY_NOTIFY_URL` 改为公网地址
3. 使用 ngrok 或反向代理暴露后端回调地址

注意：

- `ALIPAY_NOTIFY_URL` 不能使用 `localhost`

## 9. 推荐测试顺序

建议测试人员按以下顺序进行：

1. 环境启动检查
2. 登录 / 注册 / 验证码
3. 商品浏览 / 搜索 / 详情
4. 加购 / 购物车
5. 下单 / 订单列表 / 订单详情
6. mock 支付
7. 管理端商品和订单履约
8. 图片上传
9. 支付宝真实联调（如本轮要求）

## 10. 常见问题

### 10.1 `migrate` 失败

通常是因为：

- MySQL 未启动
- 测试库不存在
- `backend\.env` 中数据库账号密码错误

### 10.2 初始化数据失败

通常是因为：

- 数据库连接失败
- migrate 未完成

### 10.3 前端能开，接口全报错

优先检查：

1. 后端是否正常启动
2. 前端是否通过 5173 访问
3. 后端是否运行在 8000

### 10.4 支付回调不生效

优先检查：

1. `ALIPAY_NOTIFY_URL` 是否为公网地址
2. ngrok 是否在线
3. 后端回调地址是否可访问

## 11. 建议后续持续补齐

如果后续每个迭代都要给测试使用，建议继续补：

1. `backend/.env.test`
2. 一键创建测试数据库脚本
3. 生产级 Docker 编排（当前仅提供测试可复现版本，见第 12 章）
4. 测试报告模板
5. 已知问题模板

## 12. 迭代2容器化测试栈（Windows，本地可复现）

本章节目标：提供“测试可复现”容器化环境，不替代当前手册中的 ngrok/反向代理流程。

容器编排文件：

- `deploy/docker-compose.iter2.yml`
- `deploy/.env.iter2.example`

### 12.1 启动前准备

1. 安装 Docker Desktop（Windows），并确认 Docker Engine 已运行。
2. 打开 PowerShell，进入项目根目录。
3. 复制环境变量模板：

```powershell
cd d:\shop
copy deploy\.env.iter2.example deploy\.env.iter2
```

4. 如需真实支付回调联调，编辑 `deploy\.env.iter2`：

- `ALIPAY_NOTIFY_URL` 改为公网可访问地址（例如 ngrok 域名）
- `ALIPAY_APP_ID / ALIPAY_PRIVATE_KEY / ALIPAY_PUBLIC_KEY` 填真实值

### 12.2 一键启动与停止

启动（前端+后端+MySQL+Redis）：

```powershell
docker compose -f deploy/docker-compose.iter2.yml --env-file deploy/.env.iter2 up -d
```

查看容器状态：

```powershell
docker compose -f deploy/docker-compose.iter2.yml --env-file deploy/.env.iter2 ps
```

停止并删除容器（保留数据卷）：

```powershell
docker compose -f deploy/docker-compose.iter2.yml --env-file deploy/.env.iter2 down
```

停止并删除容器 + 数据卷（彻底重置）：

```powershell
docker compose -f deploy/docker-compose.iter2.yml --env-file deploy/.env.iter2 down -v
```

### 12.3 访问地址与端口

- 前端：`http://127.0.0.1:5173`
- 后端：`http://127.0.0.1:8000`
- MySQL：`127.0.0.1:3307`
- Redis：`127.0.0.1:6380`

以上端口可在 `deploy/.env.iter2` 中调整。

### 12.4 健康检查命令

查看服务健康状态：

```powershell
docker compose -f deploy/docker-compose.iter2.yml --env-file deploy/.env.iter2 ps
```

查看后端日志：

```powershell
docker compose -f deploy/docker-compose.iter2.yml --env-file deploy/.env.iter2 logs -f backend
```

查看前端日志：

```powershell
docker compose -f deploy/docker-compose.iter2.yml --env-file deploy/.env.iter2 logs -f frontend
```

接口连通性检查：

```powershell
Invoke-WebRequest -Uri "http://127.0.0.1:8000/api/products/categories/" -UseBasicParsing | Select-Object StatusCode
Invoke-WebRequest -Uri "http://127.0.0.1:5173/" -UseBasicParsing | Select-Object StatusCode
```

### 12.5 与 ngrok/反向代理并存说明

容器化方案和现有流程可以并存，不互相覆盖：

1. 如果只做本机测试：直接访问 `127.0.0.1:5173` + `127.0.0.1:8000`。
2. 如果要公网回调后端（支付通知）：将 ngrok 指向本机后端端口 `8000`，并更新 `ALIPAY_NOTIFY_URL`。
3. 如果要公网访问前端：可沿用你现有反向代理/Nginx，再由 ngrok 指向反向代理端口。

### 12.6 常见问题（容器化场景）

1. 端口冲突（5173/8000/3307/6380 被占用）
   - 修改 `deploy/.env.iter2` 端口后重启 compose。
2. 后端启动慢或首次安装依赖时间长
   - 首次会在容器内执行 `pip install` 与 `npm install`，属于预期。
3. 后端报数据库连接失败
   - 先确认 `docker compose ... ps` 中 mysql 为 `healthy`。
4. 前端页面打开但接口报错
   - 检查 `deploy/.env.iter2` 的 `VITE_API_BASE_URL` 是否指向 `http://127.0.0.1:8000/api/`。
