# Iter2 任务拆分 + 接口契约 + 联调门禁

## 1. 目标与范围
- 迭代目标：业务增强优先，上线评价系统、销量排行、热榜轮播、基础性能优化。
- 非上线项：秒杀仅做架构预埋，不接入正式下单支付链路。
- 集成主线：`iter2/main`。

## 2. 团队拆分（5 子代理 + 1 总代理）
- 子代理1（BE-业务）：评价域模型/API/审核流/迁移。
- 子代理2（FE-业务）：详情页评价区、发布、榜单与轮播接入。
- 子代理3（BE-性能）：缓存策略、索引优化、分页稳定、限流骨架。
- 子代理4（DevOps）：容器化测试栈与环境模板。
- 子代理5（QA）：迭代2测试脑图、用例、回归与性能基线说明。
- 总代理：冻结契约、仲裁冲突、控制合并顺序、联调与验收。

## 3. 契约冻结（并行开发前置）

### 3.1 接口清单（冻结）
1. `GET /api/products/products/top-sales/`
2. `GET /api/products/products/hot-feed/`
3. `GET /api/reviews/product/{id}/`
4. `POST /api/reviews/product/{id}/`
5. `GET /api/reviews/my/`
6. `PATCH /api/admin/reviews/{id}/audit/`

### 3.2 返回字段最小集合（冻结）
- 榜单/热榜项：`id,name,price,main_image,sales,hot_score`
- 评价项：`id,product_id,user_id,rating,content,status,created_at`

### 3.3 业务规则（冻结）
- 仅已购用户可评价。
- 默认同订单项一评。
- 榜单/热榜优先缓存读，TTL 与失效策略由 BE-性能统一管理。

## 4. 分支策略与命名
- 集成分支：`iter2/main`
- 子分支：
  - `iter2/be-reviews-ranking`
  - `iter2/fe-reviews-ranking`
  - `iter2/be-perf-baseline`
  - `iter2/devops-container`
  - `iter2/qa-test-design`

## 5. 合并顺序（固定）
1. BE-业务
2. FE-业务
3. BE-性能
4. DevOps
5. QA 文档与验收结论

## 6. 门禁规则

### 门禁1：分支与基线门
- 已有预备提交快照。
- `iter2/main` 已创建并推送。
- 基线 Tag 已打：`iter2-baseline-YYYYMMDD`。

### 门禁2：契约冻结门
- 接口路径、字段、状态码冻结。
- 冻结后不得随意破坏式修改；若调整需总代理审批并同步文档。

### 门禁3：合并门
每个子分支合并前必须满足：
- 契约检查通过（字段/状态码不漂移）。
- 迭代1核心回归通过（认证、商品、购物车、订单、支付）。
- 变更文档同步更新（本文件 + 对应模块文档）。

### 门禁4：联调验收门
- 功能验收：评价、排行、热榜轮播通过。
- 性能验收：记录并对比 P95、缓存命中率、首页首屏耗时。
- 状态标记：已实现/规划中必须准确。

## 7. 冲突处理规范
- 同一接口由单一子代理“拥有”。
- 不允许绕过总代理直接修改他组契约。
- 发生冲突时以本文件冻结规则为准。

## 8. 里程碑节奏
- 阶段A：基线与分支准备完成（门禁1）。
- 阶段B：契约冻结并行开发（门禁2 + 门禁3 + 门禁4）。

## 9. 发布口径
- 对外发布仅宣称“已实现”项。
- 秒杀与更高阶容器化能力统一标注“规划中/预埋中”。
