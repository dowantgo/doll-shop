# 架构文档

- 文档版本：v4.1
- 更新时间：2026-04-26
- 适用分支：`iter4-main`

## 1. 总体架构

项目采用前后端分离单体架构：

1. 前端：Vue 3 + Vite + Element Plus
2. 后端：Django + DRF
3. 存储：MySQL（交易主数据） + Redis（缓存、限流、会话、验证码）
4. 异步：Celery Worker + Celery Beat（支付补偿任务）
5. 第三方：支付宝沙箱、快递100（可选）

链路：

`Frontend -> /api -> Django Apps -> MySQL/Redis -> External Providers`

## 2. 业务域划分

1. `users`：注册、登录、验证码、地址、个人信息
2. `products`：商品、分类、首页 feed、热榜与销量排行
3. `cart`：购物车
4. `orders`：下单、订单状态流转、发货、物流查询
5. `payment`：支付单、回调、补偿任务
6. `reviews`：评价、敏感词过滤、回复
7. `seckill`：秒杀活动、预占、限购、并发控制
8. `coupons`：券模板、用户券、订单优惠快照
9. `refunds`：退款申请、审核、审计日志
10. `services` 层（Iter4 强化）：商品 feed 缓存、订单定价、订单生命周期治理

## 4. 关键流程

### 4.1 下单与锁券
1. 用户从购物车发起下单
2. 前端先调用价格试算接口获取应付金额
3. 用户选择优惠券并应用到待支付订单
4. 后端落订单优惠快照，保证金额可复算

### 4.2 支付与补偿
1. 创建支付单并生成二维码
2. 正常回调时更新支付单与订单状态
3. 回调缺失时，通过手工/自动补偿收敛状态

### 4.3 退款审核
1. 用户按订单项与数量发起退款
2. 系统校验可退数量与金额
3. 管理员审核同意/拒绝
4. 状态机流转：`pending -> approved/rejected -> refunding -> success/failed`

### 4.4 物流轨迹
1. 已发货订单可查询轨迹
2. Provider 成功时返回时间线
3. Provider 失败时降级提示，不阻塞主链路

## 5. 一致性与可靠性策略

1. 关键写路径使用事务与 `select_for_update`
2. 支付补偿任务可重复执行，目标是幂等收敛
3. 订单、支付、退款、库存遵循最终一致性治理
4. 后台关键操作保留审计日志
