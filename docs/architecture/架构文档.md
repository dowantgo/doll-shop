# 架构文档

- 文档版本：v4.0
- 更新时间：2026-04-25
- 适用分支：`iter4-main`

## 1. 总体架构

项目为前后端分离单体架构：
1. 前端：Vue 3 + Vite + Element Plus
2. 后端：Django + DRF
3. 存储：MySQL（交易主数据）+ Redis（缓存/限流/验证码）
4. 异步：Celery Worker + Celery Beat（支付补偿任务）
5. 第三方：支付宝、快递100（可选）

链路：
`Frontend -> /api -> Django Apps -> MySQL/Redis -> External Providers`

## 2. 业务域划分

1. `users`：认证、个人信息、地址
2. `products`：商品与分类、首页 feed
3. `cart`：购物车
4. `orders`：下单、订单状态、发货、物流查询
5. `payment`：支付交易、回调、对账补偿
6. `reviews`：评价与回复
7. `seckill`：秒杀活动与预占
8. `coupons`（Iter3）：券模板、用户券、订单优惠快照
9. `refunds`（Iter3）：退款申请、审核、审计日志
10. Iter4 性能治理层：商品 feed 缓存服务、订单定价服务、订单生命周期服务、退款聚合优化

## 3. Iter3 架构增量

## 3.1 交易引擎增强（优惠 + 秒杀协同）
1. 增加价格试算接口，支持下单前预览优惠结果。
2. 增加订单应用优惠券接口，支持待支付订单锁券。
3. 规则固定：先满减，再单券。
4. 优惠快照落库，保证可复算与可审计。

## 3.2 退款子域
1. 用户按订单项与数量发起退款。
2. 后台审核驱动状态机流转。
3. 审核动作写入审计日志。
4. 支持幂等键，避免重复提交。

## 3.3 物流适配层
1. 统一抽象 `BaseLogisticsProvider`。
2. 内置 `MockProvider`、`Kuaidi100Provider`。
3. 通过 `LOGISTICS_PROVIDER` 切换，不改业务调用层。
4. 外部失败时降级，不阻塞订单主流程。

## 3.4 支付补偿双通道
1. 手工补偿：`POST /api/pay/reconcile/`。
2. 自动补偿：Celery Beat 每 15 分钟执行 `payment.reconcile_pending`。
3. 支付单记录补偿次数、状态、错误、时间，支持追踪。

## 3.5 Iter4 性能与架构治理
1. 商品 `top-sales`、`hot-feed` 统一由 `ProductFeedService` 管理缓存 key、版本号、TTL、DB 回源与命中率统计。
2. 订单定价逻辑下沉到 `orders.services.pricing`，统一满减、优惠券、最终应付计算。
3. 订单取消、过期清理、换券等场景的锁券释放下沉到 `orders.services.lifecycle`。
4. 退款创建合并已退数量与已退金额聚合查询，减少数据库往返。
5. 对商品 feed、订单过期扫描、支付补偿、退款幂等、秒杀预占释放等高频路径补充索引。

## 4. 关键流程

## 4.1 下单与锁券
1. 用户购物车下单。
2. 前端调用试算接口获取最终应付。
3. 用户选择券并调用应用接口锁券。
4. 订单保存优惠快照。

## 4.2 支付与补偿
1. 创建支付单并进入待支付。
2. 正常回调时订单转已支付，锁券转已使用。
3. 回调缺失时，手工/自动补偿拉起状态收敛。

## 4.3 退款审核
1. 用户提交退款申请（订单项+数量）。
2. 系统校验可退数量与金额。
3. 后台审核同意/拒绝。
4. 状态流转：`pending -> approved/rejected -> refunding -> success/failed`。

## 4.4 物流轨迹
1. 已发货订单可查轨迹。
2. Provider 成功返回轨迹时间线。
3. Provider 失败返回降级提示。

## 5. 一致性与可靠性

1. 关键链路使用事务和 `select_for_update`。
2. 补偿任务可重复执行且幂等收敛。
3. 订单、支付、退款、库存四账在测试中做对齐核验。
4. 所有后台关键操作保留审计日志。

## 6. 运行配置（Iter3 关键）
