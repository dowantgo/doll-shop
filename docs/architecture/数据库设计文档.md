# 数据库设计文档

- 文档版本：v4.1
- 更新时间：2026-04-26
- 适用分支：`iter4-main`
- 数据库：MySQL 8+

## 1. 设计原则

1. 交易主账数据落 MySQL，缓存与限流落 Redis
2. 下单、支付、退款、秒杀释放等关键写路径必须在事务内完成
3. 金额字段统一使用 `Decimal(10,2)`
4. 状态字段应支持回放、审计与补偿
5. Iter4 的新增索引必须服务于真实高频查询路径，避免无效堆索引

## 2. 核心关系

1. `users_user` 1:N `orders_order`
2. `orders_order` 1:N `orders_orderitem`
3. `orders_order` 1:N `payment_paymenttransaction`
4. `orders_order` 1:1 `coupons_orderdiscountsnapshot`（可空）
5. `users_user` 1:N `coupons_usercoupon`
6. `coupons_coupontemplate` 1:N `coupons_usercoupon`
7. `orders_orderitem` 1:N `refunds_refundrequest`
8. `refunds_refundrequest` 1:N `refunds_refundauditlog`

- `pricing_snapshot`（JSON）

### 3.4 `refunds_refundrequest`

- `refund_no`（唯一）
- `user_id`

基础索引：

## 4. 现有表在 Iter3 / Iter4 的关键扩展

### 4.1 `payment_paymenttransaction`
Iter3 增加补偿审计字段：
- `reconcile_attempts`
- `last_reconcile_at`
- `last_reconcile_status`
- `last_reconcile_error`

Iter4 增加索引：
- `(payment_method, status, expire_time)`：支付补偿扫描
- `(order, status, created_at)`：订单维度支付单查询
- `(status, created_at)`：待处理支付单扫描

### 4.2 `orders_order`
关键字段：
- `payment_status`
- `payment_method`
- `paid_at`
- `shipping_company`
- `shipping_status`
- `tracking_no`
- `shipped_at`
- `delivered_at`
- `expires_at`

Iter4 增加索引：
- `(user, payment_status, status, created_at)`
- `(payment_status, status, expires_at)`
- `(tracking_no)`

### 4.3 `products_product`
Iter4 增加索引：
- `(status, sales, created_at)`：销量排行
- `(status, is_hot, hot_sort_order)`：热榜 feed

### 4.4 `refunds_refundrequest`
Iter4 增加索引：
- `(order_item, status)`：退款数量/金额聚合
- `(user, idempotency_key)`：用户维度幂等查询

### 4.5 `seckill_seckillactivity`
Iter4 增加索引：
- `(group_id, status, is_enabled)`：活动列表与分组展示
- `(status, reserved_expires_at)`：预占释放扫描

## 5. Redis 使用

1. `products:feed:version`：商品 feed 缓存版本号
2. `products:feed:stats:top-sales:*`：销量排行命中统计
3. `products:feed:stats:hot-feed:*`：热榜命中统计
4. `ratelimit:seckill_pre_reserve:*`：秒杀预占限流
5. `orders:pricing:payment-probe:user:{user_id}`：价格试算支付探针缓存（Iter4）
6. 会话、验证码、短期缓存等

说明：
- 优惠、退款、支付补偿结果均以 MySQL 为准
- Redis 不作为交易主账，只承接缓存、限流与短期探针数据

## 6. 一致性策略

1. 订单锁券与解券在事务内处理
2. 支付成功后同步更新订单与券状态
3. 退款使用幂等键与状态机防重复写入
4. 支付补偿任务以“幂等收敛”为目标，可重复执行
5. 秒杀预占与库存回补遵循事务与状态校验

## 7. Iter4 数据层收益

### 7.1 性能结果
- `top-sales` P95：`566.02ms -> 328.51ms`
- `hot-feed` P95：`511.33ms -> 272.78ms`
- `refund-create` P95：`557.44ms -> 441.51ms`
- `logistics-query` P95：`366.26ms -> 348.76ms`
- `price-preview` P95：`716.53ms -> 537.94ms`

### 7.2 数据设计收益
- 索引补齐后，高频查询路径更稳定
- 退款聚合减少数据库往返
- 支付探针缓存化降低热路径查库压力
- feed 缓存统计可直接支持性能验收与后续调优
