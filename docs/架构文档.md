# 架构文档

- 文档版本: v1.1
- 更新时间: 2026-04-09
- 适用版本: 当前主干

## 1. 架构概览

本项目采用前后端分离架构：
- 前端: Vue 3 + Vite
- 后端: Django + DRF
- 数据层: MySQL + Redis
- 支付: 支付宝网关 + 回调

```text
Browser
  -> Frontend (Vue3, Vite dev server :5173)
  -> /api/* (Axios)
  -> Django API (:8000)
  -> MySQL / Redis / Alipay Gateway
```

## 2. 分层设计

### 2.1 前端层
1. 路由层
- `frontend/src/router/index.js`
- 负责公开页、登录页、用户页、管理页路由及守卫

2. 视图层
- `frontend/src/views/*`
- 管理端页在 `frontend/src/components/admin/*`

3. 接口层
- `frontend/src/api/*`
- 按域拆分：`user/product/cart/order/payment/admin`

4. 网络基础设施
- `frontend/src/utils/request.js`
- 统一 token 注入、401 失效处理

### 2.2 后端层
1. 路由入口
- `backend/dollshop/urls.py`
- 聚合 `/api/users`、`/api/products`、`/api/cart`、`/api/orders`、`/api/admin`、`/api/pay`

2. 业务应用
- `apps/users`: 用户、地址、验证码、邮箱验证码
- `apps/products`: 分类、商品、商品图、库存日志
- `apps/cart`: 购物车
- `apps/orders`: 订单与物流
- `apps/payment`: 交易单、支付创建、状态查询、回调验签

3. 权限模型
- 默认 DRF 权限为 `IsAuthenticated`
- 公共接口在 ViewSet action 中改为 `AllowAny`
- 管理接口通过 `user.role == 'admin'` 校验

### 2.3 数据层
1. MySQL
- 业务主存储
- 订单/库存通过事务保障一致性

2. Redis
- 验证码与邮箱验证码缓存
- Django Session 缓存后端

## 3. 核心调用链路

### 3.1 登录链路
1. 获取图片验证码
2. 用户提交登录表单（用户名/邮箱 + 密码 + 验证码）
3. 后端校验验证码并认证账号
4. 下发 JWT access token
5. 前端存储 token 并带 token 访问业务接口

### 3.2 下单链路
1. 购物车提交地址创建订单
2. 后端事务中校验库存、扣减库存、生成订单项
3. 清理购物车
4. 返回订单详情

### 3.3 支付链路
1. 调用 `POST /api/pay/create_payment/`
2. 后端在事务中锁定订单并处理 `PaymentTransaction`
3. 支付宝预下单，返回 `qr_code` + `qr_code_image`
4. 前端轮询状态接口
5. 异步回调 `notify/alipay` 验签成功后更新支付状态与订单状态

## 4. 部署与联调模式

### 4.1 本地开发模式
- 前端: `localhost:5173`
- 后端: `localhost:8000`
- Vite 代理 `/api` 到后端

### 4.2 内网穿透模式（支付回调）
1. ngrok 暴露后端（或 Nginx 反向代理后暴露统一端口）
2. `.env` 中将 `ALIPAY_NOTIFY_URL` 配置为公网地址
3. 支付宝平台配置同一回调地址

### 4.3 Nginx 反向代理模式
- `/` 转发到前端
- `/api/`、`/media/`、`/static/` 转发到后端
- 适合外部演示与统一域名

## 5. 配置中心

### 5.1 后端配置
- 文件: `backend/dollshop/settings.py`
- 来源: `backend/.env`（`load_dotenv(..., override=True)`）

### 5.2 关键配置项
1. 数据库
- `DB_NAME/DB_USER/DB_PASSWORD/DB_HOST/DB_PORT`

2. Redis
- `REDIS_URL`

3. 邮件
- `EMAIL_HOST`、`EMAIL_PORT`、`EMAIL_USE_SSL/TLS`

4. 支付
- `ALIPAY_APP_ID`
- `ALIPAY_PRIVATE_KEY`
- `ALIPAY_PUBLIC_KEY`
- `ALIPAY_NOTIFY_URL`

## 6. 稳定性与一致性设计

1. 事务控制
- 下单、取消订单、支付状态落库均使用事务
- `select_for_update` 用于并发写保护

2. 缓存降级
- 验证码模块存在本地内存兜底缓存实现

3. 异常处理
- 支付模块区分配置错误、网络错误、验签错误
- API 保持结构化错误返回

## 7. 当前技术债与改进建议

1. 文案/注释存在历史编码遗留，应逐步清理为统一 UTF-8 中文。
2. API 文档路由未开放，建议补充 `/api/schema/` 与 `/api/docs/`。
3. `users.PaymentRecord` 与 `payment.PaymentTransaction` 职责重叠，建议后续收敛为单一支付账务模型。
4. 建议补充自动化测试（支付状态流转、并发库存、回调验签）。
