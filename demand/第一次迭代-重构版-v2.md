- 文档版本: v2.0
- 更新时间: 2026-04-10
- 适用项目: 玩偶商城 Doll Shop
- 关联文档:
  - `docs/架构文档.md`

  - `README.md`
- 文档定位: 第一迭代可测试需求说明，用于后续测试用例设计、回归验证、发布验收。

## 0. 本版重构说明

v1 版本已经覆盖第一迭代的主要模块，但多数子功能只描述了功能点，缺少可测试规则。

v2 版本在保留原 10 个模块的基础上，补充以下内容:

1. 全局权限、状态、错误返回、数据一致性规则。
2. 各模块的业务规则、异常流程、数据影响和验收标准。
3. 订单、支付、库存、验证码、图片上传等高风险链路的测试依据。
4. 明确“已实现、待验证、待实现、待确认”状态，避免把未确认规则误当成已完成开发。

## 1. 状态标记规则

| 状态 | 含义 |
| --- | --- |
| 已实现 | 当前代码或文档已明确支持，可进入回归测试 |
| 待验证 | 设计上应支持，但仍需测试确认 |
| 待实现 | 已纳入第一迭代或收尾需求，但当前仍需开发完成 |
| 待确认 | 业务规则尚未最终确定，测试前需产品或开发确认 |

## 2. 第一迭代目标

第一迭代目标是完成“用户可浏览商品、加购、下单、支付、查看订单，管理员可维护商品与订单履约，系统支持本地和外网支付回调联调”的最小可用闭环。

完成后应满足:

1. 用户端核心交易链路闭环可用。
2. 管理端商品和订单履约维护可用。
3. 支付宝扫码支付与异步回调链路可验证。
4. 本地开发、Nginx 反向代理、ngrok 外网回调联调流程可执行。
5. 具备首版测试用例、测试报告和发布记录。

## 3. 范围边界

### 3.1 本迭代范围

- 前端: `frontend`，Vue 3 + Vite + Element Plus。
- 后端: `backend`，Django + DRF。
- 数据层: MySQL + Redis。
- 支付: 支付宝扫码支付、mock 支付基础验证。
- 运维联调: 本地直连、Nginx 反向代理、ngrok 外网回调。
- 测试资料: 测试用例、测试报告、发布操作记录。

### 3.2 不在本迭代范围

- 推荐算法。
- 优惠券。
- 发票。
- 退款自动化。
- 复杂风控。
- 大规模性能压测。
- 微信真实支付。

### 3.3 待确认边界

1. 商品图片上传和支付二维码图片直显是否必须作为第一迭代收尾交付。
2. mock 支付是否作为正式测试范围，还是仅作为开发调试辅助。
3. 基础性能测试是否只做轻量基线。

## 4. 全局规则

### 4.1 用户与权限

1. 未登录用户只能访问公开接口，如注册、登录、验证码、商品浏览等。
2. 需要登录的用户接口必须校验 JWT access token。
3. token 缺失、过期或无效时，后端返回 401，前端应清理登录态并引导重新登录。
4. 普通用户访问管理端接口时，后端返回 403，前端应提示无权限。
5. 管理端接口必须校验 `user.role == "admin"`。
6. 用户只能查看和操作自己的购物车、地址、订单、支付记录。

状态: 待验证。

### 4.2 API 错误返回

1. API 错误应保持结构化返回，至少包含可读错误信息。
2. 业务校验失败应返回 400。
3. 未登录返回 401。
4. 无权限返回 403。
5. 资源不存在返回 404。
6. 外部服务异常，如支付宝网络错误，应返回明确错误提示，不应暴露敏感配置或堆栈。

状态: 待验证。

### 4.3 数据一致性

1. 下单、取消订单、支付状态更新必须使用事务保证一致性。
2. 涉及库存扣减或回补的操作必须避免并发超卖。
3. 支付回调应具备幂等性，重复回调不应重复修改库存、销量或订单状态。
4. 任一关键步骤失败时，应回滚同一事务内的数据变更。

状态: 待验证。

### 4.4 枚举值

订单状态:

- `pending`: 待支付
- `paid`: 已支付
- `shipped`: 已发货
- `delivered`: 已完成或已收货
- `cancelled`: 已取消

支付状态:

- `pending`: 待支付
- `paid`: 已支付
- `expired`: 已过期
- `closed`: 已关闭
- `failed`: 支付失败
- `refunded`: 已退款，本迭代不做自动退款

物流公司:

- `sf`: 顺丰
- `zto`: 中通
- `yto`: 圆通
- `sto`: 申通
- `yunda`: 韵达

物流状态:

- `not_shipped`: 未发货
- `shipped`: 已发货
- `in_transit`: 运输中
- `arrived`: 已到达
- `signed`: 已签收

状态: 待验证。

## 5. 模块 M01: 用户认证与安全校验

### M01-01 注册

业务目标:

- 用户可以通过用户名、邮箱、密码、邮箱验证码、图片验证码完成注册。

前置条件:

- 用户未登录。
- 邮箱验证码服务可用，或能明确返回发送失败原因。
- 图片验证码可正常生成。

主流程:

1. 用户获取图片验证码。
2. 用户填写用户名、邮箱、密码、图片验证码。
3. 用户请求发送邮箱验证码。
4. 用户输入邮箱验证码并提交注册。
5. 系统创建用户，默认角色为 `user`。

规则与约束:

1. 用户名必须唯一。
2. 邮箱格式必须合法。
3. 邮箱验证码有效期为 10 分钟。
4. 图片验证码有效期为 5 分钟。
5. 验证码错误或过期时，不允许注册成功。
6. 注册成功后不得直接赋予管理员权限。

异常流程:

- 用户名重复。
- 邮箱格式错误。
- 密码为空或不符合后端校验要求。
- 图片验证码错误或过期。
- 邮箱验证码错误或过期。
- 邮箱发送失败。

数据影响:

- 创建 `users_user` 记录。
- Redis 写入或读取验证码键: `captcha:{captcha_id}`、`email_code:{type}:{email}`。

验收标准:

- 注册成功后可使用该账号登录。
- 注册失败时用户数据不应落库。
- 错误提示能指出主要失败原因。

状态: 已实现，验证码稳定性待验证。

### M01-02 登录

业务目标:

- 用户通过用户名或邮箱、密码、图片验证码登录系统。

前置条件:

- 用户已注册。
- 用户账号未被禁用。

主流程:

1. 用户获取图片验证码。
2. 用户输入账号、密码、验证码。
3. 系统校验验证码和账号密码。
4. 系统下发 JWT access token。
5. 前端保存 token，并进入用户首页或原访问页面。

规则与约束:

1. 图片验证码错误时不得校验通过。
2. 密码错误时不得登录。
3. token 过期后需要重新登录。
4. 前端后续请求应自动携带 token。

异常流程:

- 账号不存在。
- 密码错误。
- 验证码错误。
- 验证码过期。
- token 过期后访问业务接口。

数据影响:

- 登录不直接修改核心业务表。
- 可能读取 Redis 验证码缓存。

验收标准:

- 登录成功后可访问购物车、订单、个人中心。
- 登录失败时不保存 token。
- token 失效后前端能统一处理 401。

状态: 已实现，待回归验证。

### M01-03 会话与鉴权

业务目标:

- 统一处理用户登录态、请求鉴权和权限异常。

规则与约束:

1. 前端请求拦截器自动注入 token。
2. 401 时清理本地 token 和用户状态。
3. 403 时保留登录态，但提示权限不足。
4. 管理页面需限制非管理员访问。

验收标准:

- 未登录访问需登录页面或接口时被拦截。
- 普通用户不能进入或调用管理后台功能。
- 管理员可以访问管理端统计、商品、订单、用户接口。

状态: 待验证。

## 6. 模块 M02: 商品浏览（用户端）

### M02-01 首页商品列表

业务目标:

- 用户可浏览上架商品、热销商品，并按分类或关键词筛选。

规则与约束:

1. 用户端只展示 `status` 为上架的商品。
2. 热销商品根据 `is_hot` 和 `hot_sort_order` 展示。
3. 分类筛选只返回对应分类商品。
4. 关键词搜索至少应匹配商品名称；是否匹配描述为待确认。
5. 无结果时前端展示空状态。
6. 库存为 0 的商品是否可展示为待确认。

异常流程:

- 分类不存在。
- 搜索关键词为空。
- 商品接口请求失败。

验收标准:

- 首页商品列表可正常加载。
- 分类筛选、关键词搜索结果正确。
- 下架商品不应出现在用户端列表。

状态: 已实现，规则细节待验证。

### M02-02 列表页加购入口

业务目标:

- 用户可在商品列表直接加入购物车。

规则与约束:

1. 未登录用户点击加购时，应提示登录并跳转登录页。
2. 已登录用户可将上架且库存充足的商品加入购物车。
3. 重复加购同一商品时，应增加购物车数量，不应创建重复购物车项。
4. 商品下架或库存不足时不得加购成功。

验收标准:

- 未登录态加购不会写入购物车。
- 已登录态加购成功后购物车数量变化正确。

状态: 已实现，库存和下架限制待验证。

## 7. 模块 M03: 商品详情页

### M03-01 详情展示

业务目标:

- 用户可查看商品主图、图集、描述、价格、库存等详情。

规则与约束:

1. 商品存在时展示完整详情。
2. 商品不存在时展示 404 或友好错误。
3. 下架商品是否允许通过详情 URL 访问为待确认。
4. 多图展示应优先展示主图，再按 `sort_order` 排序。

验收标准:

- 商品详情字段与后端数据一致。
- 商品图片加载失败时不影响页面基础信息展示。

状态: 已实现，图片排序待验证。

### M03-02 详情页加购

业务目标:

- 用户可在详情页选择数量并加入购物车。

规则与约束:

1. 数量必须为正整数。
2. 数量不得超过当前库存。
3. 库存为 0 时，应禁用加购或提示库存不足。
4. 未登录用户加购时跳转登录。

验收标准:

- 合法数量加购成功。
- 非法数量、超库存、未登录状态有明确处理。

状态: 已实现，边界待验证。

## 8. 模块 M04: 购物车与下单

### M04-01 购物车管理

业务目标:

- 登录用户可查看、修改、移除、清空购物车商品。

规则与约束:

1. 购物车项按用户隔离。
2. 同一用户同一商品只能有一条购物车项。
3. 修改数量必须为正整数。
4. 修改数量不得超过库存。
5. 商品下架或库存不足时，购物车应给出提示，是否允许继续保留为待确认。
6. 清空购物车只清空当前用户数据。

异常流程:

- 购物车项不存在。
- 商品不存在或已下架。
- 修改数量超过库存。
- 未登录访问购物车。

数据影响:

- 增删改 `cart_cartitem`。

验收标准:

- 购物车列表金额、数量、小计展示正确。
- 修改、移除、清空后数据实时刷新。

状态: 已实现，异常和库存边界待验证。

### M04-02 创建订单

业务目标:

- 用户从购物车选择收货地址生成订单。

前置条件:

- 用户已登录。
- 购物车非空。
- 已选择有效收货地址。
- 商品上架且库存充足。

主流程:

1. 用户在购物车点击结算。
2. 用户选择收货地址。
3. 后端在事务中锁定商品库存。
4. 创建订单和订单项。
5. 扣减库存，增加销量。
6. 清空购物车。
7. 返回订单详情。

规则与约束:

1. 空购物车不能创建订单。
2. 无有效地址不能创建订单。
3. 库存不足时订单创建失败，库存和购物车不应改变。
4. 订单项价格使用下单时商品价格快照。
5. 订单初始状态为 `pending`，支付状态为 `pending`。
6. 订单过期时间为创建后 30 分钟。
7. 并发下单不得导致库存为负数。

异常流程:

- 地址不存在或不属于当前用户。
- 商品库存不足。
- 商品已下架。
- 事务中任一步骤失败。

数据影响:

- 创建 `orders_order`。
- 创建 `orders_orderitem`。
- 更新 `products_product.stock`。
- 更新 `products_product.sales`。
- 清空当前用户 `cart_cartitem`。
- 是否写入 `products_inventorylog` 为待验证。

验收标准:

- 成功下单后购物车清空。
- 订单金额等于订单项快照价乘数量之和。
- 库存扣减和销量增加正确。
- 失败时不产生半成品订单。

状态: 已实现，事务和并发待验证。

## 9. 模块 M05: 个人中心与地址管理

### M05-01 个人信息展示

业务目标:

- 用户可查看当前登录账号的基础信息。

规则与约束:

1. 只展示当前登录用户信息。
2. 未登录访问应跳转登录或返回 401。
3. 普通用户不得查看其他用户信息。

验收标准:

- 页面显示用户名、邮箱、手机号、头像等已支持字段。

状态: 已实现，待验证。

### M05-02 收货地址 CRUD

业务目标:

- 用户可维护自己的收货地址，并设置默认地址。

规则与约束:

1. 地址必填字段包括收货人、手机号、省、市、区、详细地址。
2. 手机号格式必须合法。
3. 每个用户只能有一个默认地址。
4. 新增第一个地址时，是否自动设为默认地址为待确认。
5. 设置某地址为默认后，该用户其他地址应自动取消默认。
6. 用户只能编辑或删除自己的地址。
7. 删除默认地址后的默认地址处理规则为待确认。
8. 被历史订单引用的地址删除后，订单 `address_id` 可为 NULL，但订单详情应如何展示为待确认。

异常流程:

- 字段缺失。
- 手机号格式错误。
- 编辑或删除不存在的地址。
- 操作其他用户地址。

数据影响:

- 增删改 `users_address`。
- 订单地址外键删除后按数据库设计为 `SET NULL`。

验收标准:

- 地址 CRUD 正常。
- 默认地址唯一。
- 三级联动选择省市区可用。

状态: 已实现，默认地址和订单地址历史展示规则待确认。

## 10. 模块 M06: 订单列表与详情

### M06-01 订单列表

业务目标:

- 用户可查看自己的订单列表，并按状态筛选。

规则与约束:

1. 用户只能查看自己的订单。
2. 当前已明确筛选: 全部、待支付、已支付。
3. 是否需要筛选已发货、已完成、已取消为待确认。
4. 待支付订单可取消。
5. 已支付、已发货、已完成订单是否允许用户取消为待确认，建议第一迭代不允许。

验收标准:

- 订单列表展示订单号、金额、状态、创建时间。
- 筛选结果与状态一致。
- 待支付取消后状态更新正确。

状态: 已实现，状态筛选范围待确认。

### M06-02 订单详情

业务目标:

- 用户可查看订单商品行、支付方式、物流信息。

规则与约束:

1. 订单详情只允许订单所属用户或管理员查看。
2. 商品行展示名称、单价、数量、小计。
3. 支付方式展示订单当前支付方式。
4. 物流信息展示快递公司、物流状态、快递单号。
5. 商品被删除后，订单项仍应保留下单时快照信息。
6. 地址被删除后订单详情如何展示为待确认。

验收标准:

- 明细金额与订单总金额一致。
- 未发货订单展示未发货状态。
- 已发货订单展示物流字段。

状态: 已实现，历史快照展示待验证。

### M06-03 超时订单清理

业务目标:

- 待支付订单超过 30 分钟后自动清理或自动取消。

规则与约束:

1. 仅处理 `pending` 且支付状态为 `pending` 的订单。
2. 超时阈值为 30 分钟。
3. 超时后订单状态应变为 `cancelled` 或支付交易变为 `expired`，最终规则待确认。
4. 超时取消时应回补库存并回退销量。
5. 已支付订单不得被超时清理。

数据影响:

- 更新 `orders_order.status`。
- 更新 `payment_transaction.status`，如存在交易。
- 回补 `products_product.stock`。
- 回退 `products_product.sales`。

验收标准:

- 超时待支付订单不可继续支付。
- 库存和销量回补正确。
- 已支付订单不受影响。

状态: 已实现，状态落点待确认并验证。

## 11. 模块 M07: 支付链路

### M07-01 创建支付

业务目标:

- 用户为待支付订单创建支付交易单，获取支付二维码。

前置条件:

- 用户已登录。
- 订单属于当前用户。
- 订单状态为 `pending`。
- 订单支付状态为 `pending`。

主流程:

1. 用户点击支付。
2. 前端调用 `POST /api/pay/create_payment/`。
3. 后端锁定订单与待支付交易。
4. 创建或复用 `PaymentTransaction`。
5. 调用支付宝预下单。
6. 返回二维码信息。

规则与约束:

1. 已支付订单不得重复创建支付单。
2. 已取消或已超时订单不得创建支付单。
3. 同一订单重复创建支付时，复用未过期待支付交易还是创建新交易为待确认。
4. 交易金额必须等于订单金额。
5. 支付方式第一迭代以 `alipay` 为主。
6. `mock` 支付是否纳入正式测试范围为待确认。
7. `wechat` 为预留方式，第一迭代不做真实支付。
8. 返回字段应包含 `qr_code`，二维码图片直显增强后应包含 `qr_code_image`。

异常流程:

- 订单不存在。
- 订单不属于当前用户。
- 订单已支付。
- 订单已取消或过期。
- 支付宝配置错误。
- 支付宝网络错误。

数据影响:

- 创建或更新 `payment_transaction`。
- 不应在创建支付时改变订单为已支付。

验收标准:

- 合法订单返回二维码。
- 非法订单返回明确错误。
- 支付交易金额与订单金额一致。

状态: 基础链路已实现，二维码图片直显待实现或待验证。

### M07-02 支付状态更新

业务目标:

- 通过用户轮询和支付宝异步回调更新支付交易和订单状态。

规则与约束:

1. 用户可通过状态接口查询支付结果。
2. 支付宝异步回调必须验签。
3. 验签成功且金额匹配时，交易状态更新为 `paid`。
4. 订单状态更新为 `paid`，支付状态更新为 `paid`。
5. 回调重复到达时应幂等。
6. 回调金额与订单金额不一致时不得更新为已支付。
7. 验签失败时不得更新订单。
8. 支付成功后记录 `paid_at` 或 `paid_time`。

异常流程:

- 验签失败。
- 重复回调。
- 金额不一致。
- 订单不存在。
- 交易单不存在。
- 回调时订单已取消或超时。

数据影响:

- 更新 `payment_transaction.status`、`trade_no`、`paid_time`。
- 更新 `orders_order.status`、`payment_status`、`paid_at`、`trade_no`。

验收标准:

- 支付成功后订单在列表和详情中展示已支付。
- 重复回调不会重复修改或报错导致状态异常。
- 验签失败和金额错误不会污染订单状态。

状态: 已实现基础链路，异常和幂等待验证。

### M07-03 回调外网联调

业务目标:

- 通过 ngrok 或反向代理让支付宝可访问本地后端回调接口。

规则与约束:

1. `.env` 中 `ALIPAY_NOTIFY_URL` 必须配置为公网可访问地址。
2. 支付宝平台配置的回调地址应与系统配置一致。
3. `localhost` 不能作为真实支付宝回调地址。
4. 回调接口应支持公网访问并返回支付宝期望响应。

验收标准:

- 外网地址可访问后端回调接口。
- 支付宝回调能进入后端日志或状态更新逻辑。
- 回调成功后前端轮询能看到订单状态变化。

状态: 已实现，待联调记录补齐。

## 12. 模块 M08: 管理后台商品管理

### M08-01 商品列表与编辑

业务目标:

- 管理员可新增、编辑、删除商品，并维护分类、价格、库存、上下架状态。

规则与约束:

1. 仅管理员可访问。
2. 商品名称不能为空。
3. 价格必须大于等于 0。
4. 库存必须为非负整数。
5. 分类可为空，但如果选择分类，分类必须存在。
6. 下架商品不应在用户端列表展示。
7. 删除已有订单引用过的商品时，订单项商品外键可为空，但快照信息应保留。
8. 直接修改库存时是否写库存日志为待确认。

异常流程:

- 普通用户访问管理接口。
- 字段缺失。
- 价格或库存非法。
- 分类不存在。
- 删除被引用商品后的历史订单展示。

验收标准:

- 管理端商品 CRUD 可用。
- 用户端商品展示与上下架状态一致。
- 库存修改后用户端展示同步变化。

状态: 已实现，库存日志和历史订单展示待验证。

### M08-02 图片管理

业务目标:

- 管理员可上传商品图片，设置主图，管理图集，替代原 URL 字段模式。

规则与约束:

1. 支持上传图片文件，目标替代手动填写 URL。
2. 支持格式为 `.jpg`、`.jpeg`、`.png`、`.webp`，最终范围待确认。
3. 单图大小限制待确认，建议第一迭代设定明确上限。
4. 每个商品可有多张图片。
5. 同一商品最多只能有一张主图。
6. 未设置主图时，是否自动将第一张图片设为主图为待确认。
7. 删除主图后，应要求重新设置主图或自动选择下一张，规则待确认。
8. 图片排序按 `sort_order` 展示。
9. 非图片文件上传应失败并提示。

数据影响:

- 增删改 `products_productimage`。
- 图片文件写入后端 media 目录。

验收标准:

- 管理员可上传、查看、删除商品图片。
- 用户端商品列表和详情能展示主图。
- 多图展示顺序正确。
- 非法文件上传失败。

状态: 待实现或待验证，属于第一迭代收尾待确认项。

## 13. 模块 M09: 管理后台订单履约管理

### M09-01 物流字段维护

业务目标:

- 管理员可维护已支付订单的快递公司、物流状态、快递单号。

规则与约束:

1. 仅管理员可操作。
2. 未支付订单不允许发货，除非业务明确允许，默认不允许。
3. 发货时快递公司和快递单号必填。
4. 发货后订单状态应更新为 `shipped`。
5. 物流状态可从 `not_shipped` 更新为 `shipped`、`in_transit`、`arrived`、`signed`。
6. 物流状态是否允许回退为待确认，建议第一迭代不允许任意回退。
7. 用户确认收货后订单状态应更新为 `delivered`，如该功能已实现则纳入测试。

异常流程:

- 普通用户调用管理履约接口。
- 未支付订单发货。
- 快递单号为空。
- 快递公司不在枚举范围。
- 物流状态非法。

数据影响:

- 更新 `orders_order.shipping_company`。
- 更新 `orders_order.shipping_status`。
- 更新 `orders_order.tracking_no`。
- 更新 `orders_order.shipped_at`。
- 可能更新 `orders_order.status`。

验收标准:

- 已支付订单可维护物流信息。
- 未支付订单不能发货。
- 用户订单详情可看到物流信息。

状态: 已实现，状态约束待验证。

## 16. 需求决策清单

以下问题已按当前实现进度和第一迭代 MVP 边界给出决策。后续生成测试用例时，以“决策结论”为准。

| 编号 | 问题 | 决策结论 | 测试影响 |
| --- | --- | --- | --- |
| Q01 | 商品图上传和支付二维码图片直显是否必须纳入第一迭代收尾 | 纳入第一迭代收尾 | 生成商品图片上传、删除、设主图、二维码图片展示用例 |
| Q02 | 第一版 GitHub Tag 是否使用 `v1.0.0-iter1` | 使用该命名，但必须在测试通过后再打 tag | 发布验收用例只检查是否具备打 tag 条件，不在需求确认阶段打 tag |
| Q03 | 是否做基础性能基线测试 | 只做轻量性能基线，不做压力测试 | 生成少量商品列表、登录、下单、支付查询响应时间用例 |
| Q04 | 搜索是否匹配商品描述，还是只匹配商品名称 | 搜索匹配商品名称和商品描述 | 覆盖名称命中、描述命中、无结果、空关键词 |
| Q05 | 库存为 0 的商品是否允许展示、加购 | 允许展示，不允许加购或下单成功 | 覆盖库存 0 展示、加购限制、下单库存不足拦截 |
| Q06 | 删除默认地址后如何设置新的默认地址 | 第一迭代不强制自动迁移默认地址 | 测试地址 CRUD 和结算选址；默认地址自动迁移列为后续优化 |
| Q07 | 历史订单引用的地址或商品被删除后如何展示 | 允许地址或商品引用为空，但订单详情页面不得异常 | 覆盖历史地址/商品缺失时订单详情可正常展示空状态 |
| Q08 | 超时订单最终状态是 `cancelled` 还是支付交易 `expired` 组合 | 当前实现为删除超时订单并回补库存；后续建议改为取消保留 | 第一迭代测试当前库存回补行为，同时记录“保留取消订单”为优化项 |
| Q09 | 同一订单重复创建支付时复用交易还是创建新交易 | 复用未过期 pending 支付交易，并刷新支付信息 | 覆盖重复创建支付不会产生多个混乱的 pending 交易 |
| Q10 | mock 支付是否纳入正式测试范围 | 纳入开发和回归测试，不作为真实支付验收标准 | 用 mock 覆盖支付成功状态流转；真实验收仍以支付宝回调为准 |
| Q11 | 物流状态是否允许回退 | 第一迭代管理端允许维护物流状态；严格状态机后续优化 | 测试管理端可维护物流字段；不把“禁止回退”作为第一迭代硬性验收 |
| Q12 | 图片上传格式、大小、主图自动选择规则 | 格式按 `image/*` 判断；大小暂不限制；首图可由前端设为主图 | 覆盖图片文件、非图片拦截、首图主图、多图主图唯一；大小限制后续优化 |
